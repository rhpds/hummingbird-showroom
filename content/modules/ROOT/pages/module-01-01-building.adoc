[[introduction]]
== Introduction

In this sub-module, you'll move from running pre-built images to creating your own Hummingbird-based containers. You'll learn the multi-stage build pattern that uses full UBI images for compilation and Hummingbird images for runtime, achieving dramatic size reductions while maintaining functionality.

We'll also verify that the security tools (cosign, syft, grype) -- installed via xref:appendix-a-rhel-setup.adoc[Appendix A] -- are available, as you'll use them extensively in Sub-Module 1.2 for scanning, SBOM generation, and signing.

[[sample-application]]
== Create Sample Application

Let's create a simple Quarkus REST application that we'll build into a Hummingbird container.

=== Step 2: Scaffold the Project with Quarkus CLI

The Quarkus CLI generates the full project structure -- `pom.xml`, Maven wrapper, source directories, and configuration -- in a single command:

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab
cd ~/hummingbird-lab
quarkus create app com.example:sample-app \
    --extension='rest,rest-jackson' \
    --no-code
cd sample-app
----

.*What `quarkus create app` generates:*
* `pom.xml` with Quarkus BOM, `quarkus-rest` and `quarkus-rest-jackson` extensions
* Maven wrapper (`mvnw`, `.mvn/`) for reproducible builds without a system Maven installation
* `src/main/java/com/example/` and `src/main/resources/` directory structure
* Default `application.properties`
* `.gitignore` and `.dockerignore`

The `--no-code` flag tells the CLI to skip generating starter endpoint code so we can write our own.

[TIP]
====
Using the Maven wrapper (`./mvnw`) instead of a system `mvn` ensures every developer (and every container build) uses the exact same Maven version, avoiding "works on my machine" issues.
====

=== Step 2a: Prepare Project for Multi-Stage Container Builds

The Quarkus scaffold generates a `.dockerignore` that excludes everything except pre-built artifacts -- this is designed for single-stage builds where you compile locally first. Our multi-stage Containerfile compiles _inside_ the container, so source files must be included in the build context.

Replace the generated `.dockerignore`:

[source,bash,role=execute]
----
cat > .dockerignore << 'EOF'
target/
.git/
.gitignore
README.md
*.cmd
EOF
----

The Quarkus scaffold also generates files with restrictive permissions (600/700) that can cause "Permission denied" errors during container builds. Fix them:

[source,bash,role=execute]
----
chmod -R a+rX .mvn/ src/
chmod a+r pom.xml
chmod a+x mvnw
----

[NOTE]
====
The Maven wrapper's `distributionSha256Sum` in `.mvn/wrapper/maven-wrapper.properties` may contain an incorrect hash for the Maven version. If you see a "Failed to validate Maven distribution SHA-256" error during the container build, clear the hash:

[source,bash]
----
sed -i 's/distributionSha256Sum=.*/distributionSha256Sum=/' .mvn/wrapper/maven-wrapper.properties
----
====

=== Step 3: Create Application Resource

[source,bash,role=execute]
----
mkdir -p src/main/java/com/example
cat > src/main/java/com/example/GreetingResource.java << 'EOF'
package com.example;

import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;
import java.time.Instant;
import java.util.LinkedHashMap;
import java.util.Map;

@Path("/")
public class GreetingResource {

    @GET
    @Produces(MediaType.APPLICATION_JSON)
    public Map<String, String> hello() {
        Map<String, String> response = new LinkedHashMap<>();
        response.put("message", "Hello from Hummingbird!");
        response.put("runtime", "Java " + System.getProperty("java.version"));
        response.put("platform", System.getProperty("os.name").toLowerCase());
        response.put("timestamp", Instant.now().toString());
        return response;
    }

    @GET
    @Path("/health")
    @Produces(MediaType.APPLICATION_JSON)
    public Map<String, String> health() {
        Map<String, String> response = new LinkedHashMap<>();
        response.put("status", "healthy");
        return response;
    }
}
EOF
----

=== Step 4: Configure Application Properties

[source,bash,role=execute]
----
cat > src/main/resources/application.properties << 'EOF'
quarkus.http.host=0.0.0.0
quarkus.http.port=8080
EOF
----

[[multi-stage-containerfile]]
== Multi-Stage Containerfile

Now let's create the Containerfile that builds our application using the multi-stage pattern.

=== Step 5: Create Multi-Stage Containerfile

[source,bash,role=execute]
----
cat > Containerfile << 'EOF'
# Multi-stage build: UBI builder -> Hummingbird runtime

# Build arguments for registry flexibility
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=quay.io/hummingbird-hatchling

# ============================================
# Stage 1: Build stage using full UBI image
# ============================================
FROM ${BUILDER_REGISTRY}/openjdk-21:latest AS builder

# Install unzip needed by the Maven wrapper to extract the Maven distribution
USER root
RUN microdnf install -y unzip && microdnf clean all

WORKDIR /build

# Copy Maven wrapper and dependency manifest first (layer cache)
COPY mvnw pom.xml ./
COPY .mvn ./.mvn

# Download dependencies (cached unless pom.xml changes)
RUN ./mvnw dependency:go-offline -B

# Copy source and build
COPY src ./src
RUN ./mvnw package -DskipTests -B

# ============================================
# Stage 2: Runtime stage using Hummingbird
# ============================================
FROM ${RUNTIME_REGISTRY}/openjdk:21-runtime

WORKDIR /app

# Copy the Quarkus fast-jar layout
COPY --from=builder --chown=1001:1001 /build/target/quarkus-app/lib/ ./lib/
COPY --from=builder --chown=1001:1001 /build/target/quarkus-app/*.jar ./
COPY --from=builder --chown=1001:1001 /build/target/quarkus-app/app/ ./app/
COPY --from=builder --chown=1001:1001 /build/target/quarkus-app/quarkus/ ./quarkus/

# Run as non-root user
USER 1001

# Expose port
EXPOSE 8080

# JVM configuration
ENV JAVA_OPTS_APPEND="-Dquarkus.http.host=0.0.0.0"

# Start application
ENTRYPOINT ["java", "-jar", "quarkus-run.jar"]
EOF
----

.*Key Points in this Containerfile:*

* **Build Arguments**: `ARG BUILDER_REGISTRY` and `ARG RUNTIME_REGISTRY` allow switching registries without editing the file -- useful for Buildah and air-gapped environments
* **Multi-Stage**: Stage 1 (builder) has JDK 21 and the Maven wrapper; Stage 2 (runtime) has only the JRE
* **Unzip Installation**: The UBI9 OpenJDK 21 builder image does not include `unzip`, which the Maven wrapper requires to extract the Maven distribution
* **Layer Caching**: `mvnw`, `pom.xml`, and `.mvn/` are copied before source so Maven dependencies are cached across builds
* **Security**: Non-root user (1001), proper file ownership with `--chown`
* **Fast-Jar Layout**: Quarkus packages the app into `lib/`, `app/`, `quarkus/`, and `quarkus-run.jar` for optimal startup
* **Image Naming**: Hummingbird images use tags for variants -- `openjdk:21-runtime` (not `openjdk-21-runtime:latest`)

[IMPORTANT]
====
**Why Multi-Stage Builds Matter:**

**Stage 1 (Builder)**:
* Full UBI OpenJDK 21 image (~435MB) -- includes JDK compiler, Maven, and build tools
* Downloads dependencies and compiles the application
* If you shipped this image as-is with your app: ~454MB

**Stage 2 (Runtime)**:
* Hummingbird OpenJDK 21 runtime base: ~253MB (JRE only -- no compiler, no Maven, no build tools)
* Final image with app: ~272MB (app layers add ~19MB)

**Result**: ~272MB final image instead of ~454MB -- a **40% size reduction**. The savings come from stripping build tools and the JDK compiler, shipping only the JRE and your application.
====

[[building-image]]
== Building the Image

=== Step 6: Build with Podman

Build your first Hummingbird-based container image:

[source,bash,role=execute]
----
cd ~/hummingbird-lab/sample-app
podman build -t hummingbird-demo:v1 .
----

[NOTE]
====
The first build downloads Maven dependencies and may take 3-5 minutes. Subsequent builds are much faster thanks to layer caching -- only layers with changed content are rebuilt.
====

.*Expected output (excerpt):*
----
[1/2] STEP 1/9: FROM registry.access.redhat.com/ubi9/openjdk-21:latest AS builder
[1/2] STEP 2/9: USER root
[1/2] STEP 3/9: RUN microdnf install -y unzip && microdnf clean all
[1/2] STEP 4/9: WORKDIR /build
[1/2] STEP 5/9: COPY mvnw pom.xml ./
...
[INFO] BUILD SUCCESS
...
[2/2] STEP 1/10: FROM quay.io/hummingbird-hatchling/openjdk:21-runtime
...
[2/2] STEP 10/10: ENTRYPOINT ["java", "-jar", "quarkus-run.jar"]
COMMIT hummingbird-demo:v1
Successfully tagged localhost/hummingbird-demo:v1
----

[[podman-desktop-build]]
== Building with Podman Desktop GUI

Now let's build the same image using Podman Desktop's visual interface.

=== Podman Desktop Example 4: Build Your App on Top of Hummingbird

. Open Podman Desktop
. Navigate to **Images** tab
. Click **Build** button in the top-right
. In the "Build Image" dialog:
  - **Containerfile path**: Click **Browse** and select `~/hummingbird-lab/sample-app/Containerfile`
  - **Build context**: Select `~/hummingbird-lab/sample-app` (the directory containing the Containerfile)
  - **Image name**: `hummingbird-demo:gui-v1`
  - **Build arguments** (optional):
    * Add `BUILDER_REGISTRY=registry.access.redhat.com/ubi9`
    * Add `RUNTIME_REGISTRY=quay.io/hummingbird-hatchling`
. Click **Build**
. Monitor build progress in the GUI:
  - You'll see each Containerfile step execute in real-time
  - Stage 1 (builder) will show Maven download and build output
  - Stage 2 (runtime) will show the COPY operations
. Once complete, the image `hummingbird-demo:gui-v1` will appear in your **Images** list

.*Key Observation:*
The Podman Desktop build uses the same Podman engine backend as the CLI, so the result is identical. The GUI provides a visual representation of the build process, which can be helpful for learning and debugging.

While the visual editor is convenient, most developers prefer using their IDE (VS Code, IntelliJ, etc.) for Containerfile editing, as it provides better syntax highlighting and linting extensions.

=== Troubleshooting

**Issue: Build fails with Maven dependency download errors**
[source,bash]
----
# Clear build cache and rebuild
podman build --no-cache -t hummingbird-demo:v1 .

# Check network connectivity
curl https://repo.maven.apache.org/maven2/
----

**Issue: Image size larger than expected**
[source,bash]
----
# Verify multi-stage build is working
podman history hummingbird-demo:v1

# Check if builder stage artifacts leaked into runtime
podman run --rm hummingbird-demo:v1 ls /build 2>/dev/null && echo "LEAK" || echo "OK"
----

**Issue: Container fails to start**
[source,bash]
----
# Check logs for errors
podman logs demo

# Verify the fast-jar layout is intact
podman run --rm hummingbird-demo:v1 ls -la /app/

# Test Java availability in the runtime image
podman run --rm hummingbird-demo:v1 java -version
----

**Issue: Maven build fails inside the container**
[source,bash]
----
# Run builder stage interactively to debug
podman run --rm -it registry.access.redhat.com/ubi9/openjdk-21:latest bash

# Verify Maven wrapper is executable
ls -la mvnw
chmod +x mvnw
./mvnw --version
----
