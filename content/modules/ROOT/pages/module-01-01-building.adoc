[[introduction]]
== Introduction

In this sub-module, you'll move from running pre-built images to creating your own Hardened Images based containers. You'll learn the multi-stage build pattern using separate builder and runtime images, achieving dramatic size reductions while maintaining functionality.

[[types]]
== Available types of hardened images

There are three main categories of images provided in the collection of hardened images: runtimes, services, and tools. Each of these are built in the same fashion but provide different capabilities.  Runtimes include various languages like `java`, `python`, and `dotnet` and also have `-builder` variants to allow for complex application tasks. Services include network based applications like `caddy`, `memcached`, and `haproxy` for deploying minimal versions of common services. Tools include things like `git`, `curl`, and `jq` for running a known containerized version in an isolated manner like within a CI pipeline.

----

With our `index.html` ready to serve, we just need a web server. The `caddy` image can be used for this sort of simple testing by running the default container image and passing the directory as a volume mount. 

[source,bash,role=execute,subs=attributes+]
----
podman run -d --rm --name caddy-server \
  -p 8080:8080 \
  -v ~/webserver:/usr/share/caddy:ro,Z \
  {hummingbird-registry}/caddy:latest
----

[source,bash,role=execute]
----
curl http://localhost:8080
----

[source,bash,role=execute]
----
podman stop caddy-server
----

This will work for very simple testing, but to use this image in a production environment, it's better to create a custom Containerfile for your final webserver.

[source,bash,role=execute,subs=attributes+]
----
cat > ~/webserver/Containerfile << 'EOF'
FROM {hummingbird-registry}/caddy:latest

COPY index.html /usr/share/caddy/
EOF
----

We can simply add the content to be served to the default Caddy location and create the final container. If we needed more complex Caddy configurations, we can also create and add a custom Caddyfile to override the defaults in the image.

[source,bash,role=execute]
----
podman build -t my-website -f ~/webserver/Containerfile ~/webserver
----

[source,bash,role=execute]
----
podman run -d --rm --name webserver -p 8080:8080 my-website
----

[source,bash,role=execute]
----
curl http://localhost:8080
----

Network service images are fairly straightforward, but what about tools? Why would we want a containerized version of `curl` available?

On the current system, we have `curl` installed as an RPM. But, what if you were building on a system without `curl` and without the privileges to install new packages? This can be a common occurence in CI/CD runners, systems used to build software in a pipeline often with minimal packages installed.

In that case we could run the same test as we did with the local package using a known good and specific version of `curl` via `podman`

[source,bash,role=execute,subs=attributes+]
----
podman run --rm --net=host {hummingbird-registry}/curl:latest http://localhost:8080
----

[source,bash,role=execute]
----
podman stop webserver
----
Note the use of `--net=host` here to let an user container connect to the exposed port on the host. There's a wide range of possibilities here, like testing an API call with a POST request, downloading a file for local inspection, etc.

Containerized versions of common tools let's us provide a known good, common set of capabilities to any host while ensuring the same security profiles to the tools as any other hardened image. This makes keeping fleets of systems, like build or test farms, in sync since they can all use the exact same tool image instead of requiring local tools.
We can be sure we've got the right images by verifing the signature with a tool like `cosign`. We'll cover more on that later.

[[runtimes]]
=== Simple runtime examples
While services and tools can be useful, the most commonly used types of hardened images are runtimes. These provide the platform for various different languages and are typically available in different variants: eg `default`, `builder`, and `fips`. The details will vary from image to image, but the `default` variant is purpose built as a deployment target with only critical dependencies and security metadata. The `builder` variant adds to the `default` both rpm and language specific package managers to allow for installation of additional dependencies, debug and troubleshooting, and for multi-stage-containerfiles. 
The `fips` variants include specially validated OpenSSL modules for the enforcement of specific crypto algorithms required for certain regulatory regimes. We'll look at this variant in a later module.

==== Create a simple Python Flask application
We'll look at how using the UBI Python images you have today differ from the new `default` variant. To do that, we need a Python application to containerize.

Our Flask application will use the development server to serve an HTML file as a static file, which means we'll need to add it to the `static` directory in our Containerfile later.
[source,bash,role=execute]
----
cat > ~/flask/app.py << 'EOF'
from flask import Flask

app = Flask(__name__,)

@app.route("/")
def index():
    return app.send_static_file("index.html")

if __name__ == "__main__":
    # Listen on all interfaces (0.0.0.0) on port 8080
    app.run(host="0.0.0.0", port=8080)

EOF
----

Then, copy the `index.html` from the Caddy example. No need to reinvent the wheel here.
[source,bash,role=execute]
----
cp ~/webserver/index.html ~/flask/
----

We'll ship this application in two ways to explore some differences between the typical container build and one with a hardened image.

== Distroless vs distribution base images

Hardened images are what's called 'distroless' images. These styles of images are targeted builds that skip certain shells and tools typically found in standard base images. Since package managers are usually part of that missing toolset, that's how the term 'distroless' was coined. https://redhat.atlassian.net/browse/RHEL-144380

Let's build a comparison image using UBI to see the difference. Then we can see what changes we need to make to move this to a hardened `python` image. In both cases, we'll use a local mirror for python dependcies, mimicing the use of a secure enterprise index like Red Hat Trusted LIbraries.

[source,bash,role=execute]
----
# Create a UBI single stage Containerfile
cat > ~/flask/Containerfile.ubi << 'EOF'
# Stage 1: Base Image from Red Hat UBI
FROM registry.access.redhat.com/ubi9/ubi

# Install pip to manage application dependencies
RUN dnf -y install python3-pip && dnf clean all #<.>

# Create a non-root user and group for the application
# Using a different UID/GID to avoid conflict with existing users in the base image.
RUN groupadd -r -g 1005 appgroup && \
    useradd -r -u 1005 -g 1005 -d /app -s /sbin/nologin -c "Application User" appuser

# Set the working directory in the container
WORKDIR /app

# Ensure ownership is set to the new non-root user
# COPY always executes as root
COPY --chown=appuser:appgroup app.py .
COPY --chown=appuser:appgroup index.html static/

# Set environment variables for Python
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# Install application dependencies
USER root #<.>
RUN python3 -m pip install --extra-index-url http://localhost:8000 flask  #<.>

# Switch to the non-root user for runtime 
USER appuser #<.>

# Expose the port Flask will listen on
EXPOSE 8080

# Appropriately set the stop signal for the python interpreter executed as PID 1
STOPSIGNAL SIGINT #<.>
ENTRYPOINT ["python3", "./app.py"]

EOF
----
<.> Use `dnf` to install RHEL packaged `pip` for installing dependencies
<.> Install globally as `root` to skip creating a virtual environment
<.> Use `--index-url` to use our local artifact repository
<.> Reset the USER with the provided ENV variable
<.> Standard non-init container configuration

Since we're using a local PyPi mirror, we need to make sure `podman` uses thecp webserver/index.html flask/ host's networking to access mirror via `localhost`.

[source,bash,role=execute]
----
podman build --net=host -t my-flasksite:ubi -f ~/flask/Containerfile.ubi ~/flask
----

Let's do a quick check on our UBI based application and see that we're getting the results we expect.

[source,bash,role=execute]
----
podman run -d --rm --name flask-demo -p 8080:8080 my-flasksite:ubi
----

[source,bash,role=execute]
----
curl http://localhost:8080
----

[source,bash,role=execute]
----
podman stop flask-demo
----

=== Step 2: Examine the changes

The most obvious change is the base image, but there are a few other minor adjustments we need to make.

Our UBI example creates a non-root `appuser` to run the application. The hardened image already provides a default non-root user, so we can skip that step. 

This means we do need to make sure we're running the commands as the correct user for each step however. We can also skip installing `pip` as that's already included in the `default` variant for `python`.

We'll demonstrate this by installing the application dependecies globally as `root` instead of working in a virtual environment. This workflow is fine since we're only ever installing one application in this image, but your choice should make the most sense for your applications and environment.

In practice those changes give us the following Containerfile

[source,bash,role=execute,subs=attributes+]
----
cat > ~/flask/Containerfile.hi << 'EOF'
# Stage 1: Builder base image 
FROM {hummingbird-registry}/python:3.14 #<.>

# Set the working directory in the container
WORKDIR /app

# Copy the application files to the target directory
# COPY always executes as root
COPY --chown=65532 app.py . #<.>
COPY --chown=65532 index.html static/

# Set environment variables for Python
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# Switch to root from the non-root user to install dependcies
# By default, these install in /tmp/.local for the non-root user 
USER root
RUN pip install --index-url http://localhost:8000/simple flask 

# Switch to the default non-root user for runtime 
USER ${CONTAINER_DEFAULT_USER} #<.>

# Expose the port Flask will listen on
EXPOSE 8080

# Appropriately set the stop signal for the python interpreter executed as PID 1
STOPSIGNAL SIGINT
ENTRYPOINT ["python", "./app.py"]

EOF
----

<.> Change our base image to the hardened `default` image
<.> Change the user in `COPY` to the provided default UID instead of adding `appuser`
<.> Reset the USER with the provded ENV variable

Let's go ahead and build our app on the `dafault` variant.
[source,bash,role=execute]
----
podman build --net=host -t my-flasksite:hi -f ~/flask/Containerfile.hi ~/flask
----
----
STEP 7/11: RUN pip install --index-url http://localhost:8000/simple flask
error running container: from /usr/bin/crun creating container for [/bin/sh -c pip install --index-url http://localhost:8000/simple flask]: executable file `/bin/sh` not found: No such file or directory
----
What happened here? One of the changes to 'distroless' images like our new `default` is the removal of things like shells. Here `pip` needs a shell to install our application dependencies. 

Without a shell, what can we do? Switch to the `builder` variant.

Edit the Containerfile and change the `FROM` to include the appropriate tag.
[source,bash,role=execute,subs=attributes+]
----
vi ~/flask/Containerfile.hi
----
[source,dockerfile,subs=attributes+]
----
# Stage 1: Builder base image 
FROM {hummingbird-registry}/python:3.14-builder #<.>
----
<.> The tagging format for `python` images is <version>-builder

Let's retry the build using the new variant.
[source,bash,role=execute]
----
podman build --net=host -t my-flasksite:hi -f ~/flask/Containerfile.hi ~/flask
----
[source,bash,role=execute]
----
podman run -d --rm --name flask-demo -p 8080:8080 my-flasksite:hi
----

[source,bash,role=execute]
----
curl http://localhost:8080
----

Changing existing Containerfiles to use hardened images can be fairly straightforward. To help understand the default settings for hardened images, we https://gitlab.com/redhat/hummingbird/containers/-/blob/main/images/python-3-14/hummingbird/default/compat.md[provide tables] that compare them to upstream versions. 
This not only includes the properties, but also the reasoning behind any changes.

[[multi-stage]]
== Multi-stage builds with hardened images

Let's create a simple Quarkus REST application that we'll use for a more realistic multi-stage build. Multi-stage builds are the most common pattern we see when using hardeneg images. 

Multi-stage builds allow you to separate the final application artifacts from build time components and to choose the smallest runtimes for production. This keeps with the overall goal of minimal images, reduced attack and risk surfaces. Multi-stage builds provide a good blend of flexibility and control.

This will combine the use of different variants for different purposes.

[[sample-application]]
== Create sample Java application

=== Step 4: Scaffold the Project with Quarkus CLI

The Quarkus CLI generates the full project structure -- `pom.xml`, Maven wrapper, source directories, and configuration -- in a single command:

[source,bash,role=execute]
----
quarkus create app com.example:sample-app \
    --extension='rest,rest-jackson' \
    --no-code
----

.*What `quarkus create app` generates:*
* `pom.xml` with Quarkus BOM, `quarkus-rest` and `quarkus-rest-jackson` extensions
* Maven wrapper (`mvnw`, `.mvn/`) for reproducible builds without a system Maven installation
* `src/main/java/com/example/` and `src/main/resources/` directory structure
* Default `application.properties`
* `.gitignore` and `.dockerignore`

The `--no-code` flag tells the CLI to skip generating starter endpoint code so we can write our own.

[TIP]
====
Using the Maven wrapper (`./mvnw`) instead of a system `mvn` ensures every developer (and every container build) uses the exact same Maven version, avoiding "works on my machine" issues.
====

=== Step 5: Prepare Project for Multi-Stage Container Builds

The Quarkus scaffold generates a `.dockerignore` that excludes everything except pre-built artifacts -- this is designed for single-stage builds where you compile locally first. Our multi-stage Containerfile compiles _inside_ the container, so source files must be included in the build context.

[source,bash,role=execute]
----
cd sample-app
----
Replace the generated `.dockerignore`:

[source,bash,role=execute]
----
cat > .dockerignore << 'EOF'
target/
.git/
.gitignore
README.md
*.cmd
EOF
----

The Quarkus scaffold also generates files with restrictive permissions (600/700) that can cause "Permission denied" errors during container builds. Fix them:

[source,bash,role=execute]
----
chmod -R a+rX .mvn/ src/
----

[source,bash,role=execute]
----
chmod a+r pom.xml
----

[source,bash,role=execute]
----
chmod a+x mvnw
----

[NOTE]
====
The Maven wrapper's `distributionSha256Sum` in `.mvn/wrapper/maven-wrapper.properties` may contain an incorrect hash for the Maven version. If you see a "Failed to validate Maven distribution SHA-256" error during the container build, clear the hash:

[source,bash]
----
sed -i 's/distributionSha256Sum=.*/distributionSha256Sum=/' .mvn/wrapper/maven-wrapper.properties
----
====

=== Step 6: Create Application Resource

[source,bash,role=execute]

mkdir -p src/main/java/com/example
cat > src/main/java/com/example/GreetingResource.java << 'EOF'
package com.example;

import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;
import java.time.Instant;
import java.util.LinkedHashMap;
import java.util.Map;

@Path("/")
public class GreetingResource {

    @GET
    @Produces(MediaType.APPLICATION_JSON)
    public Map<String, String> hello() {
        Map<String, String> response = new LinkedHashMap<>();
        response.put("message", "Hello from Hummingbird!");
        response.put("runtime", "Java " + System.getProperty("java.version"));
        response.put("platform", System.getProperty("os.name").toLowerCase());
        response.put("timestamp", Instant.now().toString());
        return response;
    }

    @GET
    @Path("/health")
    @Produces(MediaType.APPLICATION_JSON)
    public Map<String, String> health() {
        Map<String, String> response = new LinkedHashMap<>();
        response.put("status", "healthy");
        return response;
    }
}
EOF
----

[[multi-stage-containerfile]]
== Multi-Stage Containerfile

Now let's create the Containerfile that builds our application using the multi-stage pattern.
The first stage will use the `builder` variant of the `openjdk-21` image. This provides additional tools like `dnf` to install RPMs into the environment. 
Once we've created the application with `maven`, we can copy just the necessary application files to the `runtime` variant for deployment.
Not only do all of the `maven` build artifacts stay in the first stage, but so do any addtional packages or tools we mave have needed to get a working build environment.

=== Step 8: Create Multi-Stage Containerfile

[source,bash,role=execute,subs=attributes+]
----
cat > ~/sample-app/Containerfile << 'EOF'
# Multi-stage build: builder -> runtime

# ============================================
# Stage 1: Build stage using builder variant
# ============================================
FROM {hummingbird-registry}/openjdk:21-builder AS builder # <1>

# Install unzip needed by the Maven wrapper to extract the Maven distribution
USER root
RUN dnf install -y unzip && dnf clean all # <2>

WORKDIR /build

# Copy Maven wrapper and dependency manifest first (layer cache)
COPY mvnw pom.xml ./ # <3>
COPY .mvn ./.mvn

# Download dependencies (cached unless pom.xml changes)
RUN ./mvnw dependency:go-offline -B

# Copy source and build
COPY src ./src
RUN ./mvnw package -DskipTests -B

# ============================================
# Stage 2: Runtime stage 
# ============================================
FROM {hummingbird-registry}/openjdk:21-runtime #<1>

WORKDIR /app

# Copy the Quarkus fast-jar layout
COPY --from=builder --chown=65532:65532 /build/target/quarkus-app/lib/ ./lib/ # <4>
COPY --from=builder --chown=65532:65532 /build/target/quarkus-app/*.jar ./
COPY --from=builder --chown=65532:65532 /build/target/quarkus-app/app/ ./app/
COPY --from=builder --chown=65532:65532 /build/target/quarkus-app/quarkus/ ./quarkus/

# Run as non-root user
USER 65532 # <5>

# Expose port
EXPOSE 8080

# JVM configuration
ENV JAVA_OPTS_APPEND="-Dquarkus.http.host=0.0.0.0"

# Start application
ENTRYPOINT ["java", "-jar", "quarkus-run.jar"]
EOF
----

.*Key Points in this Containerfile:*

<.> **Multi-Stage**: Stage 1 (builder) has JDK 21 and the Maven wrapper; Stage 2 (runtime) has only the JRE
<.> **Unzip Installation**: The OpenJDK 21 builder image does not include `unzip`, which the Maven wrapper requires to extract the Maven distribution
<.> **Layer Caching**: `mvnw`, `pom.xml`, and `.mvn/` are copied before source so Maven dependencies are cached across builds
<.> **Fast-Jar Layout**: Quarkus packages the app into `lib/`, `app/`, `quarkus/`, and `quarkus-run.jar` for optimal startup, we only copy those files from the first stage into the final image.
<.> **Security**: Non-root user (65532), proper file ownership with `--chown`

**Image Naming**: Hardened images use tags for variants -- `openjdk:21-runtime` (not `openjdk-21-runtime:latest`)

[[building-image]]
== Building the Image

=== Step 9: Build with Podman

Building a multi-stage image with `podman` is no different than any other image. The stages are automatically handled as part of the standard OCI specification.

[source,bash,role=execute]
----
podman build -t hummingbird-demo:v1 -f ~/sample-app/Containerfile ~/sample-app
----

[NOTE]
====
The first build downloads Maven dependencies and may take 3-5 minutes. Subsequent builds are much faster thanks to layer caching -- only layers with changed content are rebuilt.
====

.*Key Takeaways*

**Multi-Stage Build Pattern:**

* **Stage 1 (Builder)**: Hardened image (~435MB) with all build tools (JDK, Maven)
* **Stage 2 (Runtime)**: Hardened runtime (~253MB base) with only the JRE
* **Result**: ~272MB final image -- 40% smaller than shipping the full builder image (~454MB)

**Best Practices Applied:**

* Non-root user (UID 65532)
* Minimal runtime image (no build tools, no package managers, no compiler)
* Proper file ownership with `--chown`
* Dependency layer caching for faster rebuilds
* Health endpoints for orchestrator probes

=== Troubleshooting
https://redhat.atlassian.net/browse/RHEL-144380
**Issue: Build fails with Maven dependency download errors**
[source,bash]
----
# Clear build cache and rebuild
podman build --no-cache -t hummingbird-demo:v1 .

# Check network connectivity
curl https://repo.maven.apache.org/maven2/
----

**Issue: Image size larger than expected**
[source,bash]
----
# Verify multi-stage build is working
podman history hummingbird-demo:v1

# Check if builder stage artifacts leaked into runtime
podman run --rm hummingbird-demo:v1 ls /build 2>/dev/null && echo "LEAK" || echo "OK"
----

**Issue: Container fails to start**
[source,bash]
----
# Check logs for errors
podman logs demo

# Verify the fast-jar layout is intact
podman run --rm hummingbird-demo:v1 ls -la /app/

# Test Java availability in the runtime image
podman run --rm hummingbird-demo:v1 java -version
----

**Issue: Maven build fails inside the container**
[source,bash]
----
# Run builder stage interactively to debug
podman run --rm -it registry.access.redhat.com/ubi9/openjdk-21:latest bash

# Verify Maven wrapper is executable
ls -la mvnw
chmod +x mvnw
./mvnw --version
----
