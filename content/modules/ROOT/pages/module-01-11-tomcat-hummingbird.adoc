[[introduction]]
== Introduction

Hummingbird's runtime images have no package manager. You cannot `RUN dnf install tomcat` in the final stage. Everything Tomcat needs -- the JVM, the Tomcat binaries, your WAR, config -- must arrive via `COPY --from=` in a multi-stage build.

There are two viable approaches, each with different trade-offs:

[cols="1,2,2",options="header"]
|===
| Approach
| Pros
| Cons

| *RPM in builder* (Approach 1)
| Clean SBOM with proper RPM metadata; chunkah gets component boundaries; managed by `microdnf update`
| Tomcat version tied to the RPM repo (currently Tomcat 9)

| *Official image extract* (Approach 2)
| Any Tomcat version; simple extraction
| Mixed provenance (Hummingbird OS + upstream Tomcat); SBOM shows file-level, not package-level
|===

Both approaches produce a final image that is dramatically smaller than a full UBI-based Tomcat deployment, with zero CVEs from the base image.

[NOTE]
====
**Tomcat 9 vs Tomcat 10.** The RHEL 9 RPM repos ship **Tomcat 9**, which uses the `javax.servlet` namespace. If your application targets Jakarta EE 9+ (`jakarta.servlet`), you will need Approach 2 with an upstream Tomcat 10+ image. The WAR and Containerfiles in this exercise use `javax.servlet` so they work with the RPM-provided Tomcat 9.
====

=== Step 2: Create a Minimal WAR Application

For this exercise, create a simple health-check servlet. In production you would use your real WAR from a Maven or Gradle build:

[source,bash,role=execute]
----
cd ~/hummingbird-lab/tomcat-rpm

mkdir -p src/main/java/com/example src/main/webapp/WEB-INF

<?xml version="1.0" encoding="UTF-8"?>

  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>hummingbird-tomcat-demo</artifactId>
  <version>1.0.0</version>
  <packaging>war</packaging>
  <properties>
    <maven.compiler.source>21</maven.compiler.source>
    <maven.compiler.target>21</maven.compiler.target>
  </properties>
  <dependencies>
    <dependency>
      <groupId>javax.servlet</groupId>
      <artifactId>javax.servlet-api</artifactId>
      <version>4.0.1</version>
      <scope>provided</scope>
    </dependency>
  </dependencies>
</project>
EOF

cat > src/main/java/com/example/HealthServlet.java << 'EOF'
package com.example;

import javax.servlet.http.*;
import javax.servlet.annotation.WebServlet;
import java.io.IOException;

@WebServlet("/health")
public class HealthServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        resp.setContentType("application/json");
        resp.getWriter().printf(
            "{\"status\":\"UP\",\"runtime\":\"Hummingbird + Tomcat\",\"java\":\"%s\"}%n",
            System.getProperty("java.version"));
    }
}
EOF

cat > src/main/webapp/WEB-INF/web.xml << 'EOF'
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="http://xmlns.jcp.org/xml/ns/javaee"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/javaee
                             http://xmlns.jcp.org/xml/ns/javaee/web-app_4_0.xsd"
         version="4.0">
  <display-name>Hummingbird Tomcat Demo</display-name>
</web-app>
EOF
----

=== Step 3: Create the Production server.xml

The default Tomcat `server.xml` has several things you should remove or disable in a container context:

[source,bash,role=execute]
----
cd ~/hummingbird-lab/tomcat-rpm

cat > config/server.xml << 'XMLEOF'
<?xml version="1.0" encoding="UTF-8"?>
<Server port="-1" shutdown="SHUTDOWN">

  <Listener className="org.apache.catalina.startup.VersionLoggerListener" />
  <Listener className="org.apache.catalina.core.AprLifecycleListener" SSLEngine="on" />
  <Listener className="org.apache.catalina.core.JreMemoryLeakPreventionListener" />
  <Listener className="org.apache.catalina.mbeans.GlobalResourcesLifecycleListener" />
  <Listener className="org.apache.catalina.core.ThreadLocalLeakPreventionListener" />

  <Service name="Catalina">

    <!-- HTTP/1.1 only. No AJP connector (not needed in containers). -->
    <Connector
      port="8080"
      protocol="HTTP/1.1"
      connectionTimeout="20000"
      maxThreads="150"
      minSpareThreads="10"
      acceptCount="100"
      enableLookups="false"
      disableUploadTimeout="true"
      compression="on"
      compressionMinSize="2048"
      redirectPort="8443"
      server=" "
      />
    <!-- server=" " suppresses the Server header, avoiding version disclosure -->

    <Engine name="Catalina" defaultHost="localhost">

      <Host name="localhost"
            appBase="webapps"
            unpackWARs="true"
            autoDeploy="false">
        <!-- autoDeploy="false" — deploys controlled via image builds, not hotdrop -->

        <Valve className="org.apache.catalina.valves.AccessLogValve"
               directory="logs"
               prefix="access_log"
               suffix=".txt"
               pattern="%h %l %u %t &quot;%r&quot; %s %b %D"
               resolveHosts="false" />

        <Valve className="org.apache.catalina.valves.RemoteIpValve"
               remoteIpHeader="x-forwarded-for"
               protocolHeader="x-forwarded-proto" />

      </Host>
    </Engine>
  </Service>
</Server>
XMLEOF
----

.*Hardening changes from the default:*

* **`port="-1"`** on Server -- disables the shutdown port (container lifecycle is managed by the orchestrator)
* **No AJP connector** -- removed entirely; not needed when running behind an ingress controller
* **`server=" "`** -- suppresses Tomcat version in the `Server` HTTP header
* **`autoDeploy="false"`** -- deploys are controlled by image builds, not filesystem watches
* **`RemoteIpValve`** -- makes `X-Forwarded-For` trustworthy when behind a reverse proxy

=== Step 4: Create the Containerfile

[source,bash,role=execute]
----
cd ~/hummingbird-lab/tomcat-rpm

cat > Containerfile.tomcat-rpm << 'EOF'
ARG BUILDER=registry.access.redhat.com/ubi9/openjdk-21:latest
ARG RUNTIME=quay.io/hummingbird-hatchling/openjdk:21-runtime

# ── Stage 1: Build the application WAR ────────────────────────────────────────
FROM $BUILDER AS app-builder

WORKDIR /build
USER root

COPY --chown=1001:1001 pom.xml ./
RUN mvn dependency:go-offline -B
COPY --chown=1001:1001 src/ ./src/
RUN mvn package -DskipTests -B \
    && mv target/*.war target/app.war

# ── Stage 2: Assemble runtime filesystem ──────────────────────────────────────
FROM $BUILDER AS assembler

USER root
RUN microdnf install -y attr tomcat && microdnf clean all

# Create a self-contained Tomcat tree at /opt/tomcat,
# resolving the RPM symlinks so COPY works on the Hummingbird runtime.
RUN mkdir -p /opt/tomcat/{bin,conf,lib,logs,temp,webapps,work} \
    && cp /usr/share/tomcat/bin/*.jar    /opt/tomcat/bin/ \
    && cp -a /usr/share/java/tomcat/*    /opt/tomcat/lib/ \
    && cp -a /etc/tomcat/*               /opt/tomcat/conf/

COPY --from=app-builder /build/target/app.war /opt/tomcat/webapps/app.war

# Production config
COPY config/server.xml /opt/tomcat/conf/server.xml

# Tag each logical group for chunkah layer isolation
RUN setfattr -n user.component -v "tomcat-bin"  /opt/tomcat/bin \
    && setfattr -n user.component -v "tomcat-lib"  /opt/tomcat/lib \
    && setfattr -n user.component -v "tomcat-conf" /opt/tomcat/conf \
    && setfattr -n user.component -v "app-war"     /opt/tomcat/webapps/app.war

RUN chown -R 1001:1001 /opt/tomcat

# ── Stage 3: Final Hummingbird image ─────────────────────────────────────────
FROM $RUNTIME

COPY --from=assembler --chown=1001:1001 /opt/tomcat /opt/tomcat

LABEL org.opencontainers.image.title="my-tomcat-app"
LABEL org.opencontainers.image.base.name="quay.io/hummingbird-hatchling/openjdk:21-runtime"

USER 1001:1001
EXPOSE 8080

ENV CATALINA_HOME=/opt/tomcat
ENV CATALINA_BASE=/opt/tomcat

CMD ["java", \
     "-classpath", "/opt/tomcat/bin/bootstrap.jar:/opt/tomcat/bin/tomcat-juli.jar", \
     "-Dcatalina.home=/opt/tomcat", \
     "-Dcatalina.base=/opt/tomcat", \
     "-Djava.io.tmpdir=/opt/tomcat/temp", \
     "-Djava.util.logging.config.file=/opt/tomcat/conf/logging.properties", \
     "-Djava.util.logging.manager=org.apache.juli.ClassLoaderLogManager", \
     "-Djava.security.egd=file:/dev/./urandom", \
     "-Djava.awt.headless=true", \
     "-XX:+UseContainerSupport", \
     "-XX:MaxRAMPercentage=75.0", \
     "org.apache.catalina.startup.Bootstrap", "start"]
EOF
----

[NOTE]
====
**Why direct `java` invocation?** The RPM-provided Tomcat launcher (`/usr/libexec/tomcat/server`) is a bash script that sources `/usr/share/java-utils/java-functions` and other shell helpers. Rather than copying those extra dependencies into the final image, we invoke `org.apache.catalina.startup.Bootstrap` directly through the JVM. This is the same entrypoint Tomcat ultimately calls; we just skip the shell wrapper.

**Why `/opt/tomcat`?** The Tomcat RPM installs into `/usr/share/tomcat` using symlinks for `conf -> /etc/tomcat`, `lib -> /usr/share/java/tomcat`, etc. When `COPY --from=assembler` copies to the Hummingbird runtime, symlink targets outside the copy scope are lost. The assembler stage resolves everything into a single self-contained `/opt/tomcat` tree.

**Why `microdnf` instead of `dnf`?** The UBI9 openjdk-21 builder image ships with `microdnf`, not full `dnf`. Maven is pre-installed.
====

=== Step 5b: Apply chunkah Layer Splitting

After building the pre-split image, use chunkah to restructure its layers by content. chunkah reads the RPM database and `xattr` tags to place each component in its own layer:

[source,bash,role=execute]
----
cd ~/hummingbird-lab/tomcat-rpm

export CHUNKAH_CONFIG_STR="$(podman inspect localhost/my-tomcat-app:pre-split)"

podman run --rm \
  --mount=type=image,src=localhost/my-tomcat-app:pre-split,dst=/chunkah \
  -e CHUNKAH_CONFIG_STR \
  quay.io/jlebon/chunkah:v0.2.0 build \
    --max-layers 48 \
  | podman load

LOADED_ID=$(podman images --format '{{.ID}}' --filter dangling=true | head -1)
podman tag "$LOADED_ID" localhost/my-tomcat-app:v1
echo "Tagged as localhost/my-tomcat-app:v1"
----

=== Step 6: Inspect Layers

[source,bash,role=execute]
----
podman history localhost/my-tomcat-app:v1
----

You should see distinct layers for `xattr/app-war`, `xattr/tomcat-conf`, `xattr/tomcat-bin`, `xattr/tomcat-lib`, and individual RPM layers for the JVM and OS packages. This is the result of the `xattr` tagging -- chunkah places each tagged component group into its own layer.

[[tomcat-upstream]]
== Exercise 2: Tomcat from Official Image (Alternative)

When you need a different Tomcat version than what the RPM provides, you can extract the Tomcat installation from the official Docker image. This gives you mixed provenance -- Hummingbird OS layers plus upstream Tomcat binaries -- but is still dramatically cleaner than a full non-Hummingbird base.

=== Step 9: Create the Containerfile

[source,bash,role=execute]
----
cd ~/hummingbird-lab/tomcat-rpm

cat > Containerfile.tomcat-upstream << 'EOF'
ARG TOMCAT_VERSION=9.0
ARG TOMCAT_IMAGE=docker.io/library/tomcat:${TOMCAT_VERSION}-jre21
ARG BUILDER=registry.access.redhat.com/ubi9/openjdk-21:latest
ARG RUNTIME=quay.io/hummingbird-hatchling/openjdk:21-runtime

# ── Stage 1: Extract Tomcat from official image ──────────────────────────────
FROM $TOMCAT_IMAGE AS tomcat-source

# ── Stage 2: Build the application WAR ───────────────────────────────────────
FROM $BUILDER AS app-builder

WORKDIR /build
USER root

COPY pom.xml ./
RUN mvn dependency:go-offline -B
COPY src/ ./src/
RUN mvn package -DskipTests -B && mv target/*.war target/app.war

# ── Stage 3: Assemble on Hummingbird base ────────────────────────────────────
FROM $RUNTIME AS assembler

USER root

COPY --from=tomcat-source --chown=1001:1001 /usr/local/tomcat /opt/tomcat

# Remove default webapps
RUN rm -rf \
      /opt/tomcat/webapps/ROOT \
      /opt/tomcat/webapps/docs \
      /opt/tomcat/webapps/examples \
      /opt/tomcat/webapps/host-manager \
      /opt/tomcat/webapps/manager

COPY --from=app-builder --chown=1001:1001 /build/target/app.war \
     /opt/tomcat/webapps/app.war

COPY --chown=1001:1001 config/server.xml /opt/tomcat/conf/server.xml

RUN chown -R 1001:1001 \
      /opt/tomcat/logs \
      /opt/tomcat/temp \
      /opt/tomcat/work

# ── Stage 4: Final ───────────────────────────────────────────────────────────
FROM $RUNTIME

COPY --from=assembler --chown=1001:1001 /opt/tomcat /opt/tomcat

LABEL org.opencontainers.image.title="my-tomcat-app-upstream"

USER 1001:1001
EXPOSE 8080

ENV CATALINA_HOME=/opt/tomcat
ENV CATALINA_BASE=/opt/tomcat

CMD ["java", \
     "-classpath", "/opt/tomcat/bin/bootstrap.jar:/opt/tomcat/bin/tomcat-juli.jar", \
     "-Dcatalina.home=/opt/tomcat", \
     "-Dcatalina.base=/opt/tomcat", \
     "-Djava.io.tmpdir=/opt/tomcat/temp", \
     "-Djava.util.logging.config.file=/opt/tomcat/conf/logging.properties", \
     "-Djava.util.logging.manager=org.apache.juli.ClassLoaderLogManager", \
     "-Djava.security.egd=file:/dev/./urandom", \
     "-Djava.awt.headless=true", \
     "-XX:+UseContainerSupport", \
     "-XX:MaxRAMPercentage=75.0", \
     "org.apache.catalina.startup.Bootstrap", "start"]
EOF
----

.*Key differences from Approach 1:*

* The runtime user is `1001` (Hummingbird default) in both approaches
* You can pin any Tomcat minor version via the `TOMCAT_VERSION` build arg
* No RPM metadata -- the Tomcat binaries come from the upstream Docker image
* Both approaches use the same direct `java` invocation to start Tomcat

[[push-zstd]]
== Push with zstd:chunked

=== Step 12: Push the Image to a Local Registry

Start a local registry and push the image with `zstd:chunked` compression to observe the layer structure:

[source,bash,role=execute]
----
podman run -d --name registry -p 5000:5000 --replace registry:2

buildah push \
  --compression-format zstd:chunked \
  --compression-level 3 \
  --tls-verify=false \
  localhost/my-tomcat-app:v1 \
  docker://localhost:5000/my-tomcat-app:v1
----

Verify the push and inspect the layer structure:

[source,bash,role=execute]
----
skopeo inspect --tls-verify=false docker://localhost:5000/my-tomcat-app:v1 \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Layers: {len(d.get(\"Layers\",[]))}')"
----

Because of the `xattr` tagging in the Containerfile, what you get on a subsequent code-only push:

[cols="2,1",options="header"]
|===
| Layer
| Status on code-only deploy

| `app-war` (your WAR)
| Re-pulled (the WAR changed)

| `tomcat-conf` (server.xml etc.)
| Re-pulled only if config changed

| `tomcat-bin` / `tomcat-lib`
| Cached (Tomcat version unchanged)

| Hummingbird JVM/OS RPM layers
| Cached
|===

A code-only deploy on a 50-node cluster transfers only the WAR layer, not the full image.

[[tomcat-version-note]]
== Tomcat 9 vs Tomcat 10

[IMPORTANT]
====
**Servlet API namespace change.** Tomcat 10.x migrated the servlet API from `javax.servlet` to `jakarta.servlet`. The RHEL 9 RPM repos provide **Tomcat 9** (`javax.servlet`). If your application targets Jakarta EE 9+ (`jakarta.servlet`), use **Approach 2** with `TOMCAT_VERSION=10.1` and update the WAR's dependencies and `web.xml` namespace accordingly.

The WAR in this exercise uses `javax.servlet` to work with the RPM-provided Tomcat 9. To switch to Tomcat 10, change:

* `pom.xml`: `jakarta.servlet-api:6.0.0` instead of `javax.servlet-api:4.0.1`
* `HealthServlet.java`: `import jakarta.servlet.http.*` instead of `import javax.servlet.http.*`
* `web.xml`: `https://jakarta.ee/xml/ns/jakartaee` namespace, version `6.0`
* `Containerfile.tomcat-upstream`: `TOMCAT_VERSION=10.1`
====

.*Key Takeaways*

**The RPM approach is preferred** because:
* SBOM has proper RPM package metadata (not just file-level)
* chunkah gets clean RPM component boundaries for layer splitting
* Security updates arrive via `microdnf update` in the builder stage
* Single provenance chain (everything from Red Hat repos)

**The upstream approach is useful** when:
* You need a Tomcat version not yet in the RPM repos (e.g. Tomcat 10 for Jakarta EE 9+)
* You need to match an exact upstream Tomcat release
* You are migrating from an existing `tomcat:*` Docker image

**Both approaches share** the core Hummingbird advantage: zero CVEs from the base image, no package manager in the final runtime.

=== Troubleshooting

**Issue: `microdnf install tomcat` fails in the builder**
[source,bash]
----
# Check available Tomcat packages
podman run --rm --user root registry.access.redhat.com/ubi9/openjdk-21:latest \
  microdnf search tomcat
----

**Issue: Tomcat fails to start with permission errors**
[source,bash]
----
# Check the user and directory ownership
podman run --rm localhost/my-tomcat-app:v1 ls -la /opt/tomcat/logs
podman run --rm localhost/my-tomcat-app:v1 id
----

**Issue: WAR not deploying (404 on `/app/`)**
[source,bash]
----
# Check the webapps directory
podman run --rm localhost/my-tomcat-app:v1 ls /opt/tomcat/webapps/

# Check Tomcat logs
podman logs tomcat-test
----

**Issue: `ClassNotFoundException` for servlet classes**
[source,bash]
----
# This means your WAR's servlet API version doesn't match the Tomcat version.
# Tomcat 9 (RPM) uses javax.servlet; Tomcat 10 uses jakarta.servlet.
# Check which Tomcat version is installed:
podman run --rm localhost/my-tomcat-app:v1 java \
  -classpath /opt/tomcat/lib/catalina.jar \
  org.apache.catalina.util.ServerInfo
----
