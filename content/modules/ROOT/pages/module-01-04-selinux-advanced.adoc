== Introduction

This sub-module covers the advanced udica techniques that production Hummingbird deployments require.

[[device-access]]
== Device Access and --append-rules

=== The Challenge

Some containers need access to host devices -- for example, JVM-based applications may access `/dev/urandom` explicitly for entropy. The default `container_t` policy **blocks device access**, and `podman inspect` alone may not capture every permission the container needs at runtime.

The `--append-rules` flag solves this by pulling AVC denials from the audit log into the generated policy. This is the **key technique for Hummingbird**: since you cannot exercise the container interactively, you let it run a realistic workload and capture any denials.

[IMPORTANT]
====
It is **not possible** to use `audit2allow -M` with AVCs where the source context was generated by udica. Always use udica's `--append-rules` option instead.
====

=== Step 1: Prepare Directories

[source,bash,role=execute]
----
sudo mkdir -p /opt/tomcat-app/{data,logs,config}
sudo semanage fcontext -a -t container_file_t "/opt/tomcat-app(/.*)?"
sudo restorecon -Rv /opt/tomcat-app
----

=== Step 2: First Run with Device Flag

Run the container with `--device /dev/urandom` and multiple bind mounts. The `--device` flag is what causes udica to add device access rules -- without it, udica won't know to include them.

[source,bash,role=execute]
----
podman run -d \
  --name device-policy-run \
  --env container=podman \
  --device /dev/urandom \
  -v /opt/tomcat-app/data:/var/lib/app/work:rw,Z \
  -v /opt/tomcat-app/logs:/var/log/app:rw,Z \
  -v /opt/tomcat-app/config:/etc/app:ro,Z \
  -p 8080:8080 \
  hummingbird-demo:v1
----

Give it a moment to run and potentially trigger AVC denials:

[source,bash,role=execute]
----
sleep 5
podman ps -a --filter name=device-policy-run
----

=== Step 3: Generate Policy with --append-rules

Generate the policy using both `podman inspect` and the audit log:

[source,bash,role=execute]
----
podman inspect device-policy-run \
  | udica --append-rules /var/log/audit/audit.log \
          device_hummingbird
----

.*Expected output:*
----
Policy device_hummingbird with container id <id> created!

Please load these modules using:
  semodule -i device_hummingbird.cil \
    /usr/share/udica/templates/{base_container.cil,net_container.cil}
----

=== Step 4: Inspect Device Access Rules

Check what udica generated for the device access:

[source,bash,role=execute]
----
cat device_hummingbird.cil
----

The generated CIL will include the same `container_file_t` and `http_cache_port_t` rules you saw in Sub-Module 1.3, plus additional rules for device access:

----
(block device_hummingbird
    (blockinherit container)
    (blockinherit net_container)

    ; Filesystem access (same as Sub-Module 1.3)
    (allow process container_file_t ( dir ( open read getattr lock search ioctl )))
    (allow process container_file_t ( file ( getattr read lock open ioctl )))
    (allow process container_file_t ( file ( write create open append )))
    (allow process container_file_t ( dir  ( write add_name remove_name )))

    ; Device access — generated because of --device /dev/urandom
    (allow process urandom_device_t ( chr_file ( open read getattr )))

    ; Port binding
    (allow process http_cache_port_t ( tcp_socket ( name_bind )))
)
----

The `urandom_device_t` rule is the key addition -- it allows the container to read from `/dev/urandom` without granting broader device access.

=== Step 5: Load and Apply

[source,bash,role=execute]
----
sudo semodule -i device_hummingbird.cil \
  /usr/share/udica/templates/{base_container.cil,net_container.cil}

podman stop device-policy-run && podman rm device-policy-run

podman run -d \
  --name demo-device \
  --env container=podman \
  --security-opt label=type:device_hummingbird.process \
  --device /dev/urandom \
  -v /opt/tomcat-app/data:/var/lib/app/work:rw,Z \
  -v /opt/tomcat-app/logs:/var/log/app:rw,Z \
  -v /opt/tomcat-app/config:/etc/app:ro,Z \
  -p 8080:8080 \
  hummingbird-demo:v1
----

Verify the custom policy is active:

[source,bash,role=execute]
----
podman inspect demo-device --format '{{.ProcessLabel}}'
----

.*Expected output:*
----
system_u:system_r:device_hummingbird.process:s0:c<X>,c<Y>
----

=== Step 6: Cleanup

[source,bash,role=execute]
----
podman stop demo-device && podman rm demo-device
----

[[iteration-workflow]]
== Iteration Workflow for Production

Even with `--append-rules`, you may hit AVC denials in production that weren't triggered during the policy-generation run. The iteration loop for Hummingbird is slightly different from standard containers because you cannot `exec` in to reproduce issues.

=== Step 7: Create the Iteration Script

Save this script for ongoing policy refinement:

[source,bash,role=execute]
----
cat > ~/hummingbird-lab/iterate-policy.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail

CONTAINER_NAME="${1:?Usage: iterate-policy.sh <container-name> <policy-name>}"
POLICY_NAME="${2:?}"
AUDIT_LOG="/var/log/audit/audit.log"

echo "=== Checking for new AVC denials for $CONTAINER_NAME ==="

PROCESS_LABEL=$(podman inspect "$CONTAINER_NAME" \
  --format '{{.ProcessLabel}}' 2>/dev/null || echo "")

if [ -z "$PROCESS_LABEL" ]; then
  echo "Container not running or no process label found"
  exit 1
fi

SELINUX_TYPE=$(echo "$PROCESS_LABEL" | cut -d: -f3)
echo "Container SELinux type: $SELINUX_TYPE"

DENIALS=$(ausearch -m AVC -ts recent 2>/dev/null \
  | grep "scontext=.*${SELINUX_TYPE}" \
  | grep "denied" \
  | head -20 || true)

if [ -z "$DENIALS" ]; then
  echo "No new AVC denials found — policy is sufficient"
  exit 0
fi

echo ""
echo "New denials found:"
echo "$DENIALS" | audit2why 2>/dev/null || echo "$DENIALS"

echo ""
echo "=== Regenerating policy with new rules ==="

podman inspect "$CONTAINER_NAME" \
  | udica --append-rules "$AUDIT_LOG" "${POLICY_NAME}_v2"

echo ""
echo "=== Diff: what changed? ==="
if [ -f "${POLICY_NAME}.cil" ]; then
  diff "${POLICY_NAME}.cil" "${POLICY_NAME}_v2.cil" || true
fi

echo ""
echo "=== To apply the updated policy: ==="
echo "  sudo semodule -r $POLICY_NAME"
echo "  sudo semodule -i ${POLICY_NAME}_v2.cil \\"
echo "    /usr/share/udica/templates/{base_container.cil,net_container.cil}"
echo "  podman restart $CONTAINER_NAME"
SCRIPT

chmod +x ~/hummingbird-lab/iterate-policy.sh
----

=== Step 8: Understand the Workflow

The iteration workflow is:

1. **Run** container with the current custom policy
2. **Monitor** for new AVC denials: `ausearch -m AVC -ts recent | grep denied`
3. **Regenerate** policy with `--append-rules` (creates `policy_v2.cil`)
4. **Review the diff** between old and new policy to understand what changed
5. **Load** new policy: `sudo semodule -r policy && sudo semodule -i policy_v2.cil ...`
6. **Restart** container

[TIP]
====
In production, run this script on a schedule or as part of your monitoring pipeline. New denials typically appear within the first hours of deployment as the application exercises all its code paths.
====

[[policy-distribution]]
== Policy Storage and Distribution

=== Repository Structure

The `.cil` file belongs in version control alongside your Containerfile:

----
repo/
  Containerfile
  selinux/
    myapp_hummingbird.cil          # committed, reviewed, versioned
    load-policy.sh                 # helper for node provisioning
  k8s/
    deployment.yaml                # references the SELinux type
----

=== Kubernetes Deployment with SELinux

In your Kubernetes/OpenShift deployment, the pod spec references the policy by name:

[source,yaml]
----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  template:
    spec:
      containers:
        - name: app
          image: registry.example.com/apps/myapp:v1.2.0
          securityContext:
            seLinuxOptions:
              type: myapp_hummingbird.process
          volumeMounts:
            - name: config
              mountPath: /app/config
              readOnly: true
            - name: logs
              mountPath: /app/logs
      volumes:
        - name: config
          hostPath:
            path: /opt/myapp/config
            type: Directory
        - name: logs
          hostPath:
            path: /opt/myapp/logs
            type: Directory
----

The `seLinuxOptions.type` must match the policy name loaded via `semodule`.

=== Cluster-Wide Distribution

For OpenShift, use a **MachineConfig** to load the `.cil` policy on every node during provisioning. For vanilla Kubernetes, a **DaemonSet** that runs `semodule -i` on each node is the standard pattern.

=== Troubleshooting

**Issue: `--append-rules` produces empty policy**
[source,bash]
----
ausearch -m AVC -ts recent | head -20

sudo systemctl restart auditd
----

**Issue: Device access still denied after policy load**
[source,bash]
----
ausearch -m AVC -ts recent | grep denied | grep urandom

podman inspect container \
  | udica --append-rules /var/log/audit/audit.log policy_v2
----

**Issue: `semodule -r` fails (policy not found)**
[source,bash]
----
sudo semodule -l | grep your_policy

sudo semodule -r your_policy 2>/dev/null; \
sudo semodule -i your_policy.cil /usr/share/udica/templates/*.cil
----
