[[introduction]]
== Introduction

**Cloud Native Buildpacks** provide "source-to-image" capabilities without requiring Dockerfiles. They auto-detect application types (Node.js, Python, Java, Go, etc.) and build optimized images using best practices.

In this sub-module, you'll create a ClusterBuildStrategy that combines Buildpacks' convenience with Hummingbird's minimal, hardened runtime images, getting the best of both worlds: zero-config builds with production-grade security.

[[buildpacks-strategy]]
== Create Hummingbird Buildpacks Strategy

=== Step 3: Create ClusterBuildStrategy

Create a custom ClusterBuildStrategy that uses Buildpacks with Hummingbird runtimes:

[source,yaml,role=execute]
----
cat << 'EOF' | oc apply -f -
apiVersion: shipwright.io/v1beta1
kind: ClusterBuildStrategy
metadata:
  name: buildpacks-hummingbird
spec:
  parameters:
  - name: BUILDER_IMAGE
    description: The Buildpacks builder image
    default: "paketobuildpacks/builder-jammy-base:latest"
  - name: RUN_IMAGE
    description: The runtime image for the final container
    default: "paketobuildpacks/run-jammy-base:latest"
  - name: SOURCE_SUBPATH
    description: A subpath within the source directory
    default: "."
  - name: ENV_VARS
    description: Environment variables for the build
    type: array
    defaults: []
  volumes:
  - name: layers-dir
    emptyDir: {}
  - name: cache-dir
    emptyDir: {}
  - name: home-dir
    emptyDir: {}
  steps:
  - name: prepare
    image: registry.access.redhat.com/ubi9/ubi-minimal:latest
    command:
    - /bin/bash
    args:
    - -c
    - |
      set -e
      chown -R 1000:1000 /tekton/home /layers /cache
    volumeMounts:
    - name: layers-dir
      mountPath: /layers
    - name: cache-dir
      mountPath: /cache
    - name: home-dir
      mountPath: /tekton/home
  - name: build-and-push
    image: $(params.BUILDER_IMAGE)
    imagePullPolicy: Always
    command:
    - /cnb/lifecycle/creator
    args:
    - -app=$(params.shp-source-root)/$(params.SOURCE_SUBPATH)
    - -run-image=$(params.RUN_IMAGE)
    - -cache-dir=/cache
    - -layers=/layers
    - -report=/layers/report.toml
    - -uid=1000
    - -gid=1000
    - $(params.shp-output-image)
    env:
    - name: CNB_PLATFORM_API
      value: "0.13"
    - name: HOME
      value: /tekton/home
    volumeMounts:
    - name: layers-dir
      mountPath: /layers
    - name: cache-dir
      mountPath: /cache
    - name: home-dir
      mountPath: /tekton/home
    securityContext:
      runAsUser: 1000
      runAsGroup: 1000
  - name: results
    image: registry.access.redhat.com/ubi9/ubi-minimal:latest
    command:
    - /bin/bash
    args:
    - -c
    - |
      set -e
      cat /layers/report.toml | grep "digest" | cut -d'"' -f2 | cut -d'"' -f2 | tr -d '\n' > $(results.shp-image-digest.path)
    volumeMounts:
    - name: layers-dir
      mountPath: /layers
EOF
----

[NOTE]
====
The default `RUN_IMAGE` is the Paketo run image. When a Hummingbird Node.js runtime becomes available in the https://quay.io/organization/hummingbird-hatchling[hummingbird-hatchling] organization, override this parameter to use it (e.g., `quay.io/hummingbird-hatchling/nodejs-20:latest`).
====

.*Expected output:*
----
clusterbuildstrategy.shipwright.io/buildpacks-hummingbird created
----

.*What this strategy does:*

* Uses **Paketo Buildpacks** builder image to auto-detect application type
* Configures runtime images for the final container via `RUN_IMAGE` parameter (swap to Hummingbird when available)
* Runs builds as **non-root user** (1000) with a `prepare` step to set directory ownership
* Sets `HOME=/tekton/home` so Tekton credential injection works for the non-root build user
* Extracts image digest via a `results` step for traceability

[NOTE]
====
**Buildpacks Lifecycle Phases:**

1. **DETECTING**: Identifies application type (Node.js, Python, Java, etc.)
2. **ANALYZING**: Checks for cached layers from previous builds
3. **BUILDING**: Installs dependencies, compiles code
4. **EXPORTING**: Creates final image layers with Hummingbird runtime
====

----

[[create-build]]
== Create Build Resource

=== Step 5: Create Build

Create a Build resource pointing to a sample Node.js application:

[source,yaml,role=execute,subs="attributes"]
----
cat << 'EOF' | oc apply -f -
apiVersion: shipwright.io/v1beta1
kind: Build
metadata:
  name: sample-nodejs-app
  namespace: hummingbird-builds
spec:
  source:
    type: Git
    git:
      url: https://github.com/shipwright-io/sample-nodejs
      revision: main
  strategy:
    name: buildpacks-hummingbird
    kind: ClusterBuildStrategy
  paramValues:
  - name: RUN_IMAGE
    value: "paketobuildpacks/run-jammy-base:latest"
  - name: SOURCE_SUBPATH
    value: "source-build"
  output:
    image: {workshop-registry}/{registry-user}/sample-nodejs:latest
    credentials:
      name: registry-credentials
EOF
----

.*Expected output:*
----
build.shipwright.io/sample-nodejs-app created
----

.*What this Build defines:*

* **Source**: Git repository with Node.js application (app is under `source-build/` subdirectory)
* **Strategy**: Use `buildpacks-hummingbird` ClusterBuildStrategy
* **Runtime**: Paketo run image (swap to Hummingbird runtime when available)
* **Output**: Push to `{workshop-registry}/{registry-user}/sample-nodejs:latest`
* **Credentials**: Use `registry-credentials` secret for push

[[execute-build]]
== Execute BuildRun

=== Step 6: Trigger a BuildRun

Execute the build by creating a BuildRun resource:

[source,yaml,role=execute]
----
cat << 'EOF' | oc apply -f -
apiVersion: shipwright.io/v1beta1
kind: BuildRun
metadata:
  name: sample-nodejs-app-run-1
  namespace: hummingbird-builds
spec:
  build:
    name: sample-nodejs-app
EOF
----

.*Expected output:*
----
buildrun.shipwright.io/sample-nodejs-app-run-1 created
----

=== Step 7: Monitor BuildRun Progress

Watch the build execution:

[source,bash,role=execute]
----
oc get buildrun sample-nodejs-app-run-1 -w
----

.*Expected output:*
----
NAME                        SUCCEEDED   REASON      STARTTIME   COMPLETIONTIME
sample-nodejs-app-run-1     Unknown     Pending     2s
sample-nodejs-app-run-1     Unknown     Running     15s
sample-nodejs-app-run-1     True        Succeeded   2m30s       5s
----

Press `Ctrl+C` once you see `Succeeded`.

[TIP]
====
Typical build times:
- **First build**: 2-4 minutes (pulls builder and runtime images, no cache)
- **Subsequent builds**: 1-2 minutes (leverages cached layers)
====

[[verify-image]]
== Verify Built Image

=== Step 9: Get Image Digest

Confirm the image was pushed successfully:

[source,bash,role=execute]
----
# Get the image digest
oc get buildrun sample-nodejs-app-run-1 -o jsonpath='{.status.output.digest}'
echo ""
----

.*Expected output:*
----
sha256:abc123def456789...
----

[TIP]
====
Verify the image was pushed to Quay via the Docker v2 registry API:

[source,bash,role=execute,subs="attributes"]
----
QUAY_ROUTE=$(oc get route -n {quay-namespace} -o jsonpath='{.items[0].spec.host}')
TOKEN=$(curl -sk "https://${QUAY_ROUTE}/v2/auth?service=${QUAY_ROUTE}&scope=repository:{registry-user}/sample-nodejs:pull" \
  -u "{registry-user}:{registry-password}" | python3 -c 'import json,sys; print(json.load(sys.stdin)["token"])')
curl -sk -H "Authorization: Bearer $TOKEN" \
  "https://${QUAY_ROUTE}/v2/{registry-user}/sample-nodejs/tags/list" | python3 -m json.tool
----

.*Expected output:*
----
{
    "name": "workshopuser/sample-nodejs",
    "tags": [
        "latest"
    ]
}
----

The `latest` tag should appear, confirming the BuildRun successfully pushed the image.

NOTE: The Quay repository is created as **private** by default. The v2 registry API is used here instead of the Quay REST API because it supports standard Docker token authentication without requiring an OAuth application token.
====

=== Step 10: Inspect Image Layers and Labels

Use `skopeo` to examine the built image and confirm it uses the Hummingbird runtime:

[source,bash,role=execute,subs="attributes"]
----
skopeo inspect docker://{workshop-registry}/{registry-user}/sample-nodejs:latest 2>/dev/null \
  | jq '.' \
  || echo "Note: skopeo not available - image verification requires registry access"
----

[NOTE]
====
The image labels should reference the Hummingbird runtime. The layer count will be lower than a typical UBI-based image because Hummingbird strips unnecessary packages. This is the same `skopeo inspect` workflow you'd use in Module 1 to verify local builds.
====

[[deploy-application]]
== Deploy Built Image

=== Step 11: Deploy to OpenShift

Deploy your Hummingbird-built application. First, link the registry credentials to the `default` ServiceAccount so pods can pull the image from the private Quay repository:

[source,bash,role=execute]
----
oc secrets link default registry-credentials --for=pull -n hummingbird-builds
----

Now create the deployment, service, and route:

[source,bash,role=execute,subs="attributes"]
----
# Create deployment
oc create deployment sample-nodejs \
  --image={workshop-registry}/{registry-user}/sample-nodejs:latest \
  -n hummingbird-builds

# Expose service
oc expose deployment sample-nodejs --port=8080 -n hummingbird-builds

# Create route
oc expose service sample-nodejs -n hummingbird-builds
----

[NOTE]
====
The `oc secrets link default registry-credentials --for=pull` step is required because Quay creates repositories as **private** by default (`CREATE_PRIVATE_REPO_ON_PUSH: true`). Without linking the pull secret to the `default` ServiceAccount, pods will fail with `ImagePullBackOff`.
====

=== Step 12: Get Route URL

[source,bash,role=execute]
----
oc get route sample-nodejs -o jsonpath='{.spec.host}'
echo ""
----

.*Expected output:*
----
sample-nodejs-hummingbird-builds.apps.cluster-abc123.example.com
----

=== Step 13: Test the Application

Test your deployed application:

[source,bash,role=execute]
----
ROUTE=$(oc get route sample-nodejs -o jsonpath='{.spec.host}')
curl http://$ROUTE
----

.*Expected output:*
----
Hello, World!
----

[TIP]
====
If you get a timeout, the application may still be starting. Check pod status:

[source,bash]
----
oc get pods -l app=sample-nodejs
oc logs -l app=sample-nodejs
----
====

[[verify-runtime]]
== Verify Runtime Image

=== Step 14: Confirm Image Size

Verify the built image in the Quay console:

[source,bash,role=execute,subs="attributes"]
----
echo "Check image at: {workshop-registry-console}/repository/{registry-user}/sample-nodejs"
----

[NOTE]
====
With the default Paketo run image, the compressed image size will be ~125-150 MB. When a Hummingbird Node.js runtime image becomes available, swapping it in via the `RUN_IMAGE` parameter will significantly reduce this because:

1. Buildpacks detected Node.js and installed only production dependencies
2. The `RUN_IMAGE` parameter swaps in the minimal Hummingbird runtime
3. No build tools or package managers in the final image
====

=== Troubleshooting

**Issue: BuildRun fails with registry push error**
[source,bash]
----
# Verify secret exists and is linked
oc get secret registry-credentials -n hummingbird-builds
oc describe sa pipeline -n hummingbird-builds | grep registry-credentials

# Recreate secret if needed (use your on-cluster Quay route)
oc delete secret registry-credentials -n hummingbird-builds
QUAY_ROUTE=$(oc get route -n quay -l quay-operator/quayregistry=quay-registry -o jsonpath='{.items[0].spec.host}')
oc create secret docker-registry registry-credentials \
  --docker-server="${QUAY_ROUTE}" \
  --docker-username=YOUR_USER \
  --docker-password=YOUR_PASSWORD \
  -n hummingbird-builds
oc secrets link pipeline registry-credentials -n hummingbird-builds
----

**Issue: BuildRun stuck in Pending**
[source,bash]
----
# Check Tekton pipeline pods
oc get pods -n openshift-pipelines

# Check events
oc get events -n hummingbird-builds --sort-by=.metadata.creationTimestamp

# Describe buildrun for details
oc describe buildrun sample-nodejs-app-run-1 -n hummingbird-builds
----

**Issue: BuildRun fails with BLOB_UPLOAD_INVALID**

This typically indicates a problem with the NooBaa object storage backend used by Quay. Check the backing store status:

[source,bash]
----
# Check NooBaa backing store status
oc get backingstore -n openshift-storage
oc get noobaa -n openshift-storage -o jsonpath='{.items[0].status.phase}'

# If backing store shows Rejected/ALL_NODES_OFFLINE, wait for recovery
oc get pods -n openshift-storage

# Once NooBaa returns to Ready/OPTIMAL, delete the failed BuildRun and retry
oc delete buildrun sample-nodejs-app-run-1 -n hummingbird-builds
# Then re-create the BuildRun (Step 6)
----

**Issue: Deployment pod stuck in ImagePullBackOff**

The on-cluster Quay registry creates repositories as **private** by default. The `default` ServiceAccount needs pull credentials:

[source,bash]
----
oc secrets link default registry-credentials --for=pull -n hummingbird-builds
oc delete pod -l app=sample-nodejs -n hummingbird-builds
----

**Issue: Application not responding**
[source,bash]
----
# Check pod status
oc get pods -l app=sample-nodejs -n hummingbird-builds

# View pod logs
oc logs -l app=sample-nodejs -n hummingbird-builds

# Check route
oc get route sample-nodejs -n hummingbird-builds
----
