[[introduction]]
== Introduction

In Sub-Module 1.1, you built a Quarkus JVM application using the multi-stage pattern: full UBI builder in Stage 1, minimal Hummingbird runtime in Stage 2. That pattern is **universal** -- it works for any language. Only the builder image and the copy commands change.

This optional exercise walks through production-ready Containerfiles for five additional scenarios across Node.js, Java (native), Python, and Go. Each one uses the same `ARG BUILDER_REGISTRY` / `ARG RUNTIME_REGISTRY` pattern for registry flexibility, and produces a dramatically smaller final image compared to building on a full UBI base.

[[nodejs-express]]
== Exercise 1: Node.js Express Application

Node.js is one of the most common container workloads. The multi-stage pattern separates `npm ci` (build dependencies) from the minimal runtime.

=== Step 1: Create the Application

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab/nodejs-express
cd ~/hummingbird-lab/nodejs-express

cat > package.json << 'EOF'
{
  "name": "hummingbird-node-demo",
  "version": "1.0.0",
  "main": "server.js",
  "scripts": { "start": "node server.js" },
  "dependencies": { "express": "^4.18.2" }
}
EOF

cat > server.js << 'EOF'
const express = require("express");
const app = express();
const PORT = process.env.PORT || 8080;

app.get("/", (req, res) => {
  res.json({
    message: "Hello from Hummingbird!",
    runtime: `Node.js ${process.version}`,
    platform: process.platform,
    timestamp: new Date().toISOString()
  });
});

app.get("/health", (req, res) => {
  res.json({ status: "healthy" });
});

app.listen(PORT, () => console.log(`Server running on port ${PORT}`));
EOF
----

=== Step 2: Create the Containerfile

[source,bash,role=execute]
----
cd ~/hummingbird-lab/nodejs-express

cat > Containerfile << 'EOF'
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=quay.io/hummingbird-hatchling

# Stage 1: Build with full UBI Node.js (npm, build tools)
FROM ${BUILDER_REGISTRY}/nodejs-20:latest AS builder

WORKDIR /opt/app-root/src

COPY --chown=1001:1001 package*.json ./
RUN npm install --omit=dev

COPY --chown=1001:1001 . .

# Stage 2: Runtime on Hummingbird Node.js
FROM ${RUNTIME_REGISTRY}/nodejs:20

WORKDIR /app

COPY --from=builder --chown=1001:1001 /opt/app-root/src/node_modules ./node_modules
COPY --from=builder --chown=1001:1001 /opt/app-root/src/server.js ./
COPY --from=builder --chown=1001:1001 /opt/app-root/src/package.json ./

USER 1001

EXPOSE 8080

CMD ["node", "server.js"]
EOF
----

=== Step 3: Build and Test

[source,bash,role=execute]
----
cd ~/hummingbird-lab/nodejs-express
podman build -t hummingbird-node-demo:v1 .

podman run -d --name node-demo -p 8081:8080 hummingbird-node-demo:v1
sleep 3
curl http://localhost:8081/
curl http://localhost:8081/health

podman images | grep hummingbird-node-demo
----

.*Expected output:*
----
{"message":"Hello from Hummingbird!","runtime":"Node.js v20...","platform":"linux","timestamp":"..."}
{"status":"healthy"}

localhost/hummingbird-node-demo  v1  ...  ~246 MB
----

Cleanup:

[source,bash,role=execute]
----
podman stop node-demo && podman rm node-demo
----

.*Key Points:*

* `npm install --omit=dev` skips devDependencies (test frameworks, linters) for a smaller runtime
* The Hummingbird `nodejs:20` runtime has the Node.js interpreter but no npm, no build tools, no compilers
* The UBI builder images run as user 1001 -- use `--chown=1001:1001` on COPY and `/opt/app-root/src` as the WORKDIR
* The final image is significantly smaller than the full UBI Node.js image (~614 MB)

[[quarkus-native]]
== Exercise 2: Quarkus Native Build

A GraalVM native executable produces the **smallest possible Java final image** -- no JVM at all. The resulting binary runs directly on `ubi-micro`.

=== Step 4: Copy the Sample App and Create the Containerfile

This exercise reuses the Quarkus sample-app from Sub-Module 1.1 and compiles it to a native binary using the Mandrel builder image (Red Hat's GraalVM distribution for Quarkus).

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab/quarkus-native
cp -r ~/hummingbird-lab/sample-app/pom.xml \
      ~/hummingbird-lab/sample-app/src \
      ~/hummingbird-lab/sample-app/.mvn \
      ~/hummingbird-lab/sample-app/mvnw \
      ~/hummingbird-lab/quarkus-native/

cd ~/hummingbird-lab/quarkus-native

cat > Containerfile << 'EOF'
ARG BUILDER_IMAGE=quay.io/quarkus/ubi-quarkus-mandrel-builder-image:jdk-21
ARG RUNTIME_REGISTRY=registry.access.redhat.com/ubi9

# Stage 1: Build native executable with Mandrel (GraalVM distribution)
FROM ${BUILDER_IMAGE} AS builder

WORKDIR /build

COPY --chown=quarkus:quarkus pom.xml ./
COPY --chown=quarkus:quarkus .mvn .mvn
COPY --chown=quarkus:quarkus mvnw ./
RUN chmod +x mvnw && ./mvnw dependency:go-offline -B

COPY --chown=quarkus:quarkus src ./src
RUN ./mvnw package -Pnative -DskipTests -B

# Stage 2: Runtime on ubi-micro (minimal base for native binaries)
FROM ${RUNTIME_REGISTRY}/ubi-micro:latest

WORKDIR /app

COPY --from=builder /lib64/libz.so.1 /lib64/libz.so.1
COPY --from=builder --chown=1001:1001 /build/target/*-runner ./application

USER 1001

EXPOSE 8080

ENTRYPOINT ["./application", "-Dquarkus.http.host=0.0.0.0"]
EOF
----

=== Step 5: Build and Test

[NOTE]
====
Native compilation takes ~2-4 minutes and requires 4GB+ of memory. The Mandrel builder image is ~1.5 GB. Be patient on the first build.
====

[source,bash,role=execute]
----
cd ~/hummingbird-lab/quarkus-native
podman build -t hummingbird-quarkus-native:v1 .
----

Run and test:

[source,bash,role=execute]
----
podman run -d --name quarkus-native-demo -p 8085:8080 hummingbird-quarkus-native:v1
sleep 2
curl http://localhost:8085/
curl http://localhost:8085/health

podman images | grep hummingbird-quarkus-native
----

.*Expected output:*
----
{"message":"Hello from Hummingbird!","runtime":"Java 21...","platform":"linux","timestamp":"..."}
{"status":"healthy"}

localhost/hummingbird-quarkus-native  v1  ...  ~79 MB
----

Check the startup time in the logs:

[source,bash,role=execute]
----
podman logs quarkus-native-demo 2>&1 | head -5
----

.*Expected output:*
----
...Quarkus... started in 0.036s. Listening on: http://0.0.0.0:8080
----

Cleanup:

[source,bash,role=execute]
----
podman stop quarkus-native-demo && podman rm quarkus-native-demo
----

.*Key Points:*

* **Mandrel** is Red Hat's GraalVM distribution optimised for Quarkus native builds
* The final image contains **only** the binary + `libz.so.1` on `ubi-micro` -- no JVM, no class files, no Maven
* Native compilation is slow (~2-4 min) but the resulting image starts in **milliseconds** (0.036s observed)
* The native binary needs `libz.so.1` -- copy it from the builder stage into `ubi-micro`
* At ~79 MB, this is 87% smaller than the JVM variant (~621 MB full UBI)

[TIP]
====
**When to choose Native vs JVM:**

* **Native**: Cold-start latency matters (serverless, scale-to-zero), smallest possible image
* **JVM**: Build speed matters, need full reflection/dynamic proxy support, debugging with standard Java tools
====

[[python-gunicorn]]
== Exercise 3: Python with gunicorn

Python applications often depend on C extensions (psycopg2 for PostgreSQL, cryptography for TLS). These need `gcc` and header files at build time but not at runtime. The multi-stage pattern separates the two.

=== Step 6: Create the Application

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab/python-gunicorn/src
cd ~/hummingbird-lab/python-gunicorn

cat > requirements.txt << 'EOF'
flask==3.0.0
gunicorn==21.2.0
EOF

cat > src/__init__.py << 'EOF'
EOF

cat > src/main.py << 'EOF'
from flask import Flask, jsonify
import platform
import sys
from datetime import datetime, timezone

app = Flask(__name__)

@app.route("/")
def hello():
    return jsonify({
        "message": "Hello from Hummingbird!",
        "runtime": f"Python {sys.version}",
        "platform": platform.system(),
        "timestamp": datetime.now(timezone.utc).isoformat()
    })

@app.route("/health")
def health():
    return jsonify({"status": "healthy"})
EOF

cat > gunicorn.conf.py << 'EOF'
bind = "0.0.0.0:8000"
workers = 2
accesslog = "-"
EOF
----

=== Step 7: Create the Containerfile

[source,bash,role=execute]
----
cd ~/hummingbird-lab/python-gunicorn

cat > Containerfile << 'EOF'
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=quay.io/hummingbird-hatchling

# Stage 1: Build with full UBI Python (pip, compilers, headers)
FROM ${BUILDER_REGISTRY}/python-311:latest AS builder

WORKDIR /opt/app-root/src

COPY --chown=1001:1001 requirements.txt ./
RUN pip install --no-cache-dir \
    --prefix=/opt/app-root/install \
    -r requirements.txt

# Stage 2: Runtime on Hummingbird Python
FROM ${RUNTIME_REGISTRY}/python:3.11

WORKDIR /app

COPY --from=builder --chown=1001:1001 /opt/app-root/install /usr/local

COPY --chown=1001:1001 src/ ./src/
COPY --chown=1001:1001 gunicorn.conf.py ./

USER 1001

EXPOSE 8000

CMD ["python3", "-m", "gunicorn", \
     "--config", "gunicorn.conf.py", \
     "src.main:app"]
EOF
----

=== Step 8: Build and Test

[source,bash,role=execute]
----
cd ~/hummingbird-lab/python-gunicorn
podman build -t hummingbird-python-gunicorn:v1 .

podman run -d --name python-gunicorn-demo -p 8083:8000 hummingbird-python-gunicorn:v1
sleep 3
curl http://localhost:8083/
curl http://localhost:8083/health

podman images | grep hummingbird-python-gunicorn
----

.*Expected output:*
----
{"message":"Hello from Hummingbird!","runtime":"Python 3.11...","platform":"Linux","timestamp":"..."}
{"status":"healthy"}

localhost/hummingbird-python-gunicorn  v1  ...  ~123 MB
----

Cleanup:

[source,bash,role=execute]
----
podman stop python-gunicorn-demo && podman rm python-gunicorn-demo
----

.*Key Points:*

* `--prefix=/opt/app-root/install` isolates pip packages into a directory that can be copied cleanly to the runtime stage
* The UBI Python builder runs as user 1001 -- use `/opt/app-root/src` as WORKDIR and `--chown=1001:1001`
* The Hummingbird `python:3.11` image uses `python3` (not `python`) as the binary name
* gunicorn is the production WSGI server (never use Flask's dev server in production)
* The Hummingbird Python runtime has the interpreter but no pip, no gcc, no headers

[[python-fastapi]]
== Exercise 4: Python FastAPI with Pre-Compiled Wheels

The **wheel pattern** is an alternative to `--prefix` that pre-compiles all dependencies into wheel archives, then installs them in the runtime stage without needing a compiler.

=== Step 9: Create the Application

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab/python-fastapi/app
cd ~/hummingbird-lab/python-fastapi

cat > requirements.txt << 'EOF'
fastapi==0.115.0
uvicorn[standard]==0.30.6
EOF

cat > app/__init__.py << 'EOF'
EOF

cat > app/main.py << 'EOF'
from fastapi import FastAPI
import platform
import sys
from datetime import datetime, timezone

app = FastAPI(title="Hummingbird FastAPI Demo")

@app.get("/")
def hello():
    return {
        "message": "Hello from Hummingbird!",
        "runtime": f"Python {sys.version}",
        "platform": platform.system(),
        "timestamp": datetime.now(timezone.utc).isoformat()
    }

@app.get("/health")
def health():
    return {"status": "healthy"}
EOF
----

=== Step 10: Create the Containerfile

[source,bash,role=execute]
----
cd ~/hummingbird-lab/python-fastapi

cat > Containerfile << 'EOF'
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=quay.io/hummingbird-hatchling

# Stage 1: Build wheels
FROM ${BUILDER_REGISTRY}/python-311:latest AS builder

WORKDIR /opt/app-root/src

COPY --chown=1001:1001 requirements.txt .

RUN pip wheel --no-cache-dir --wheel-dir /opt/app-root/wheels -r requirements.txt

# Stage 2: Runtime on Hummingbird Python
FROM ${RUNTIME_REGISTRY}/python:3.11

WORKDIR /app

COPY --from=builder /opt/app-root/wheels /opt/app-root/wheels
RUN python3 -m pip install --user --no-cache-dir --no-index \
    --find-links=/opt/app-root/wheels /opt/app-root/wheels/*

COPY --chown=1001:1001 app/ ./app/

USER 1001

ENV PATH="/tmp/.local/bin:${PATH}"

EXPOSE 8000

CMD ["python3", "-m", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
EOF
----

=== Step 11: Build and Test

[source,bash,role=execute]
----
cd ~/hummingbird-lab/python-fastapi
podman build -t hummingbird-python-fastapi:v1 .

podman run -d --name fastapi-demo -p 8084:8000 hummingbird-python-fastapi:v1
sleep 3
curl http://localhost:8084/
curl http://localhost:8084/health

podman images | grep hummingbird-python-fastapi
----

.*Expected output:*
----
{"message":"Hello from Hummingbird!","runtime":"Python 3.11...","platform":"Linux","timestamp":"..."}
{"status":"healthy"}

localhost/hummingbird-python-fastapi  v1  ...  ~162 MB
----

Cleanup:

[source,bash,role=execute]
----
podman stop fastapi-demo && podman rm fastapi-demo
----

.*Key Points:*

* `pip wheel` compiles all dependencies into `.whl` files in the builder stage
* `--no-index --find-links=` installs from local wheels only -- no network access needed in the runtime stage
* The Hummingbird Python image's `site-packages` is read-only; use `pip install --user` which installs to `/tmp/.local`
* Add `/tmp/.local/bin` to `PATH` so `uvicorn` and other scripts are found
* uvicorn is the production ASGI server for FastAPI

[NOTE]
====
**When to choose gunicorn vs uvicorn:**

* **gunicorn**: WSGI apps (Flask, Django), synchronous request handling
* **uvicorn**: ASGI apps (FastAPI, Starlette), async/await, WebSocket support
====

[[go-static]]
== Exercise 5: Go Static Binary

Go compiles to a **single static binary** with no runtime dependencies. Combined with `ubi-micro`, this produces the smallest possible Hummingbird image -- just the binary on a near-empty base.

=== Step 12: Create a Sample Go App

[source,bash,role=execute]
----
mkdir -p ~/hummingbird-lab/go-app
cat > ~/hummingbird-lab/go-app/main.go << 'GOAPP'
package main

import (
	"encoding/json"
	"fmt"
	"net/http"
	"os"
	"runtime"
	"time"
)

type Response struct {
	Message   string `json:"message"`
	Runtime   string `json:"runtime"`
	Platform  string `json:"platform"`
	Timestamp string `json:"timestamp"`
}

func main() {
	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		resp := Response{
			Message:   "Hello from Hummingbird!",
			Runtime:   fmt.Sprintf("Go %s", runtime.Version()),
			Platform:  runtime.GOOS,
			Timestamp: time.Now().UTC().Format(time.RFC3339),
		}
		w.Header().Set("Content-Type", "application/json")
		json.NewEncoder(w).Encode(resp)
	})

	http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
		w.Header().Set("Content-Type", "application/json")
		w.Write([]byte(`{"status":"healthy"}`))
	})

	port := os.Getenv("PORT")
	if port == "" {
		port = "8080"
	}
	fmt.Printf("Server running on port %s\n", port)
	http.ListenAndServe(":"+port, nil)
}
GOAPP

cat > ~/hummingbird-lab/go-app/go.mod << 'GOMOD'
module hummingbird-go-demo

go 1.21
GOMOD
----

=== Step 13: Create the Containerfile

[source,bash,role=execute]
----
cat > ~/hummingbird-lab/go-app/Containerfile << 'EOF'
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=registry.access.redhat.com/ubi9

# Stage 1: Build with UBI Go toolset
FROM ${BUILDER_REGISTRY}/go-toolset:latest AS builder

WORKDIR /opt/app-root/src

COPY --chown=1001:1001 go.mod go.sum* ./
RUN go mod download 2>/dev/null || true

COPY --chown=1001:1001 . .
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -ldflags="-s -w" -o /opt/app-root/app .

# Stage 2: Runtime on ubi-micro (minimal base for static binaries)
FROM ${RUNTIME_REGISTRY}/ubi-micro:latest

WORKDIR /app

COPY --from=builder --chown=1001:1001 /opt/app-root/app ./application

USER 1001

EXPOSE 8080

ENTRYPOINT ["./application"]
EOF
----

* No runtime, no interpreter, no package manager in the final image

[[the-universal-pattern]]
== The Universal Pattern

Every Containerfile in this exercise follows the same structure:

----
ARG BUILDER_REGISTRY=registry.access.redhat.com/ubi9
ARG RUNTIME_REGISTRY=quay.io/hummingbird-hatchling

# Stage 1: Full UBI builder image
FROM ${BUILDER_REGISTRY}/<language-toolchain>:latest AS builder
WORKDIR /build
COPY <dependency-manifest> .
RUN <install-dependencies>
COPY <source> .
RUN <compile-or-package>

# Stage 2: Minimal Hummingbird runtime
FROM ${RUNTIME_REGISTRY}/<runtime>:latest
WORKDIR /app
COPY --from=builder --chown=1001:1001 <artifact> .
USER 1001
EXPOSE <port>
CMD [<entrypoint>]
----

The only things that change between languages are:

* **Builder image**: `nodejs-20`, `openjdk-21`, `python-311`, `go-toolset`, `graalvm-ce` (UBI builders run as user 1001 -- use `--chown=1001:1001` on COPY and `/opt/app-root/src` as WORKDIR)
* **Runtime image**: `nodejs:20`, `openjdk:21-runtime`, `python:3.11`, `ubi-micro` (Hummingbird images use `<name>:<version>` format)
* **Dependency install**: `npm install --omit=dev`, `mvn dependency:go-offline`, `pip install`, `go mod download`
* **Build command**: (none for Node.js), `mvn package`, `pip wheel`, `go build`
* **Artifact copied**: `node_modules + source`, `.jar files`, `wheel packages`, `static binary`

The `ARG BUILDER_REGISTRY` / `ARG RUNTIME_REGISTRY` pattern makes every Containerfile work identically with Buildah on RHEL, Podman Desktop, or in CI/CD pipelines -- and allows easy switching to internal registry mirrors for air-gapped environments.

.*Key Takeaways*

**The Pattern is Universal:**
* Stage 1: Full UBI builder with all build tools
* Stage 2: Minimal Hummingbird runtime with only the artifact
* `ARG BUILDER_REGISTRY` / `ARG RUNTIME_REGISTRY` for portability
* Works for every language Red Hat supports

**Choose Your Runtime:**
* `ubi-micro` -- for static binaries (Go, Quarkus Native) and the smallest possible images
* `openjdk:21-runtime` -- for JVM applications
* `python:3.11` -- for Python applications
* `nodejs:20` -- for Node.js applications

**Production Patterns:**
* Pre-compile C extensions as wheels (Python)
* Use the Quarkus fast-jar layout (Java JVM)
* Always build with `CGO_ENABLED=0` for Go static binaries
* Strip debug symbols with `-ldflags="-s -w"` (Go)
