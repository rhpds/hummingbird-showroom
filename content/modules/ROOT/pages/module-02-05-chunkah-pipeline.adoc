[[introduction]]
== Introduction

In Sub-Module 1.5, you used chunkah locally to split Hummingbird images into content-based layers. At platform scale, layer splitting should happen **automatically** as part of CI/CD so that every production image is optimised without developer intervention.

This sub-module creates a Tekton Task for chunkah and inserts it into the build pipeline between the image build and vulnerability scan stages. The result: every image pushed to the registry has content-based layers and `zstd:chunked` compression.

[[tekton-chunkah-task]]
== Exercise 1: Apply the chunkah-split Tekton Task

=== Step 1: Create the Task

[source,yaml,role=execute,subs=attributes]
----
cat << 'EOF' | oc apply -f -
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: chunkah-split-task
  namespace: hummingbird-builds-{user}
  labels:
    app.kubernetes.io/part-of: hummingbird-workshop
spec:
  description: >-
    Applies content-based layer splitting to a built image using chunkah.
    Produces a new image with optimised layers for better pull deduplication,
    pushed with zstd:chunked compression.
  params:
    - name: SOURCE_IMAGE
      type: string
      description: "The image to split (fully qualified registry reference)"
    - name: OUTPUT_IMAGE
      type: string
      description: "Where to push the split image"
    - name: MAX_LAYERS
      type: string
      description: "Maximum layers chunkah will produce"
      default: "48"
    - name: CHUNKAH_IMAGE
      type: string
      description: "chunkah container image"
      default: "quay.io/jlebon/chunkah:v0.2.0"
  results:
    - name: SPLIT_IMAGE_DIGEST
      description: "Digest of the chunkah-split image"
    - name: ORIGINAL_LAYERS
      description: "Layer count of the original image"
    - name: SPLIT_LAYERS
      description: "Layer count after splitting"

  volumes:
    - name: container-storage
      emptyDir: {}

  stepTemplate:
    volumeMounts:
      - name: container-storage
        mountPath: /var/lib/containers
    securityContext:
      privileged: true

  steps:
    - name: pull-source
      image: registry.redhat.io/rhel9/podman:latest
      script: |
        #!/bin/bash
        set -euo pipefail
        echo "=== Pulling source image: $(params.SOURCE_IMAGE) ==="
        podman pull --tls-verify=false "$(params.SOURCE_IMAGE)"

        ORIG_LAYERS=$(podman inspect "$(params.SOURCE_IMAGE)" \
          --format '{{len .RootFS.Layers}}')
        echo "Original layer count: $ORIG_LAYERS"
        echo -n "$ORIG_LAYERS" > $(results.ORIGINAL_LAYERS.path)

    - name: run-chunkah
      image: registry.redhat.io/rhel9/podman:latest
      script: |
        #!/bin/bash
        set -euo pipefail

        SOURCE="$(params.SOURCE_IMAGE)"
        echo "=== Running chunkah content-based layer split ==="
        echo "    Source:     $SOURCE"
        echo "    Max layers: $(params.MAX_LAYERS)"

        export CHUNKAH_CONFIG_STR="$(podman inspect "$SOURCE")"

        podman run --rm \
          --mount=type=image,src="$SOURCE",dst=/chunkah \
          -e CHUNKAH_CONFIG_STR \
          "$(params.CHUNKAH_IMAGE)" build \
            --max-layers "$(params.MAX_LAYERS)" \
          | podman load

        LOADED_ID=$(podman images --format '{{.ID}}' --filter dangling=true | head -1)
        podman tag "$LOADED_ID" "$SOURCE"

        SPLIT_LAYERS=$(podman inspect "$SOURCE" \
          --format '{{len .RootFS.Layers}}')
        echo "Split layer count: $SPLIT_LAYERS"
        echo -n "$SPLIT_LAYERS" > $(results.SPLIT_LAYERS.path)

    - name: push-split-image
      image: registry.redhat.io/rhel9/podman:latest
      script: |
        #!/bin/bash
        set -euo pipefail

        OUTPUT="$(params.OUTPUT_IMAGE)"
        echo "=== Pushing split image with zstd:chunked ==="

        podman tag "$(params.SOURCE_IMAGE)" "$OUTPUT"

        podman push \
          --tls-verify=false \
          --compression-format zstd:chunked \
          --compression-level 3 \
          --digestfile /tmp/push-digest \
          "$OUTPUT" \
          "docker://$OUTPUT"

        DIGEST=$(cat /tmp/push-digest)

        echo "Pushed: $OUTPUT"
        echo "Digest: $DIGEST"
        echo -n "$DIGEST" > $(results.SPLIT_IMAGE_DIGEST.path)
EOF
----

.*Expected output:*

----

[[pipeline-integration]]
== Exercise 2: Pipeline Integration

The chunkah Task fits into the build pipeline **after** the image build and **before** the vulnerability scan. This ensures the scan runs against the final optimised image.

=== Pipeline Flow

----
[Git Clone] -> [Buildah Build]
  |
  v
[chunkah Split]           <-- NEW: content-based layer restructuring
  |
  v
[Grype Scan]              <-- scans the split image
  |
  v
[Cosign Sign] -> [Deploy]
----

=== Step 8: Pipeline YAML (Reference)

In a full pipeline, chunkah is inserted between the build and scan tasks:

[source,yaml]
----
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: hummingbird-build-pipeline
  namespace: hummingbird-builds-{user}
spec:
  params:
    - name: IMAGE_REGISTRY
      type: string
    - name: IMAGE_NAME
      type: string
    - name: GIT_SHA
      type: string

  tasks:
    - name: git-clone
      taskRef:
        name: git-clone
        kind: ClusterTask

    - name: buildah-build
      taskRef:
        name: buildah-podman-task
      runAfter: [git-clone]

    - name: chunkah-split
      taskRef:
        name: chunkah-split-task
      runAfter: [buildah-build]
      params:
        - name: SOURCE_IMAGE
          value: "$(tasks.buildah-build.results.IMAGE_URL)"
        - name: OUTPUT_IMAGE
          value: "$(params.IMAGE_REGISTRY)/$(params.IMAGE_NAME):$(params.GIT_SHA)-split"
        - name: MAX_LAYERS
          value: "48"

    - name: grype-scan
      taskRef:
        name: grype-scan-task
      runAfter: [chunkah-split]
      params:
        - name: IMAGE
          value: "$(tasks.chunkah-split.results.SPLIT_IMAGE_DIGEST)"

    - name: cosign-sign
      taskRef:
        name: cosign-sign-attest-task
      runAfter: [grype-scan]
----

[NOTE]
====
The vulnerability scan runs on the **split** image, not the original. Since chunkah only rearranges content without modifying it, the scan results are identical -- but the image in the registry is already optimised for pulls.
====

[[verify-optimization]]
== Exercise 3: Verify Layer Optimisation

=== Step 9: Compare Layer Counts

Use `skopeo inspect` to compare the original and split images in the registry:

[source,bash,role=execute,subs="attributes"]
----
echo "=== Original image ==="
skopeo inspect "docker://{quay_hostname}/{quay_user}/secure-nodejs:latest" --username {quay_user} --password {quay_password} 2>/dev/null \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Layers: {len(d.get(\"Layers\",[]))}')" \
  || echo "Check Quay console for layer count"

echo ""
echo "=== After chunkah ==="
skopeo inspect "docker://{quay_hostname}/{quay_user}/secure-nodejs:latest-chunked" --username {quay_user} --password {quay_password} 2>/dev/null \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Layers: {len(d.get(\"Layers\",[]))}')" \
  || echo "Check Quay console for layer count"
----

.*Expected result:*
The split image will have more layers than the original. With Hummingbird base images (which already have 30+ fine-grained layers), the increase is modest (e.g. 34 to 37) because chunkah restructures by RPM package groupings. The total compressed size is similar. The benefit appears on subsequent pulls when only changed layers need to be fetched.

[[options-reference]]
== Options Reference

[cols="2,3,1",options="header"]
|===
| Option | Description | Default

| `--max-layers N`
| Maximum layers to produce. Fewer = coarser grouping but fewer round trips. More = finer grouping but more layer overhead.
| 64 (max 448)

| `--config PATH`
| OCI image config from file
| --

| `--config-str` / `$CHUNKAH_CONFIG_STR`
| OCI image config inline (use `podman inspect` output)
| --

| `--annotation KEY=VALUE`
| Add annotation to output image
| --

| `--label KEY=VALUE`
| Add label to output image
| --

| `-v`
| Verbose output (logs to stderr)
| off

| `-vv`
| Trace output (for chunkah development)
| off
|===

*Environment variables:*

* `RUST_LOG=chunkah=debug` -- fine-grained log control
* `SOURCE_DATE_EPOCH=N` -- Unix timestamp for reproducible layer mtimes

[[known-limitations]]
== Known Limitations

As of chunkah v0.2.0:

* **OSTree-based images** (created by rpm-ostree or ostree container encapsulate) are not supported
* **Buildah versions before v1.44** require `-v $(pwd):/run/src --security-opt=label=disable`
* **Layer annotations** don't persist through the `Containerfile.splitter` flow -- use the `podman load` approach if you need annotation metadata
* **`--jobs` option** cannot be used alongside the `FROM oci-archive:` trick
* **Docker** does not support `FROM oci-archive:` -- use the `podman load` approach or skopeo for Docker environments

[IMPORTANT]
====
**Pin the chunkah image version in CI/CD pipelines.** chunkah is pre-1.0 and moving fast -- breaking changes may land at any time. The Task definition above defaults to `quay.io/jlebon/chunkah:v0.2.0` rather than `:latest` for this reason. When upgrading, test the new version in a non-production pipeline first and update the `CHUNKAH_IMAGE` parameter explicitly.
====

.*Key Takeaways*

**chunkah as a Pipeline Stage:**
* Every production image gets content-based layers automatically
* No developer intervention required -- platform team configures once
* Scanning runs on the final split image (identical content, optimised structure)

**Pull Traffic Reduction:**
* chunkah (build time): Only changed package layers need re-pulling
* zstd:chunked (pull time): Within changed layers, only changed file chunks are fetched
* Together: A single-package RPM update triggers a small delta pull across the entire cluster

**Production Best Practices:**
* Set `--max-layers 48` for a good balance between granularity and overhead
* Always push with `--compression-format zstd:chunked` for maximum benefit
* Enable `enable_partial_images = true` in `containers.conf` on pull nodes

=== Troubleshooting

**Issue: TaskRun fails with permission errors**
[source,bash,subs=attributes]
----
oc get taskrun chunkah-split-nodejs -o yaml | grep -A5 status

oc adm policy add-scc-to-user privileged -z pipeline -n hummingbird-builds-{user}
----

**Issue: chunkah image pull fails in cluster**
[source,bash,subs=attributes]
----
oc get pods -n hummingbird-builds-{user}

podman pull quay.io/jlebon/chunkah:v0.2.0
----

**Issue: zstd:chunked push fails**
[source,bash]
----
podman --version

podman push --help | grep compression
